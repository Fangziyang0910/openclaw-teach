# OpenClaw 前三部分课程大纲

## 第一部分：认识 OpenClaw

### 一、从聊天窗口到真实工作流

#### 1. 大众已经熟悉的 AI 使用方式

* **通用 AI 助手**：ChatGPT、Claude、DeepSeek、Kimi、通义千问等工具，已经被广泛用于写作、总结、翻译、学习、办公和代码辅助。
* **内容生成**：AI 可以帮助用户写文案、改通知、生成日报周报、整理课程材料和输出结构化文本。
* **信息处理**：AI 可以帮助用户总结文章、提炼会议纪要、解释复杂概念和归纳资料重点。
* **辅助思考**：AI 可以帮助用户头脑风暴、整理思路、生成方案和比较不同选择。
* **技术辅助**：AI 可以帮助用户修改代码、生成脚本、解释报错和设计自动化处理思路。

#### 2. 对话型 AI 的典型使用流程

* **打开聊天窗口**：用户进入一个 AI 对话界面。
* **输入任务材料**：用户输入问题，或粘贴文档、聊天记录、会议内容和项目背景。
* **等待模型生成**：AI 根据输入内容生成回答、总结、方案、代码或文案。
* **人工检查修改**：用户结合实际场景判断结果是否可用，并进行必要调整。
* **复制回工作现场**：用户把结果转移回文档、表格、飞书、项目系统、代码仓库或其他工作环境。

#### 3. 真实工作中的使用断点

* **上下文搬运**：文件、聊天记录、会议纪要、项目背景和历史资料需要人工复制给 AI。
* **结果转移**：AI 生成的内容需要用户手动放回飞书、文档、表格、项目系统或业务流程中。
* **任务持续性弱**：多数 AI 对话停留在单次问答，难以长期监听、定时处理、自动跟进和持续协作。
* **工作现场分散**：真实工作分布在消息入口、本地文件、浏览器、内部系统、代码仓库和自动化脚本之间。
* **工具调用受限**：用户想让 AI 读取文件、调用接口、执行脚本、发送消息或更新系统时，仍然需要大量人工操作。

#### 4. 真实工作流对 AI 的新要求

* **入口要求**：AI 需要出现在用户原本使用的工作入口中。
* **上下文要求**：AI 需要理解文件、消息、历史资料、项目背景和用户偏好。
* **工具要求**：AI 需要调用 API、脚本、数据库、网页服务和业务系统。
* **执行要求**：AI 需要围绕目标完成多步骤任务。
* **交付要求**：AI 需要把结果返回到文档、表格、消息入口或业务流程中。

### 二、Agent：从回答问题到执行任务

#### 1. Agent 的基本概念

* **目标导向**：Agent 是围绕用户目标推进任务的 AI 系统。
* **工具使用**：Agent 可以调用文件、网页、命令行、API、数据库和其他外部能力。
* **过程推进**：Agent 可以把复杂任务拆成多个步骤，并根据中间结果继续行动。
* **结果交付**：Agent 可以把最终结果返回到消息入口、文档、表格、系统或其他指定位置。
* **通俗理解**：Agent 可以理解为“会使用工具并能推进任务的 AI 助手”。

#### 2. Agent 的核心能力

* **目标理解**：识别用户真正想完成的任务。
* **任务拆解**：将复杂任务拆成多个可执行步骤。
* **工具选择**：判断当前任务需要使用哪些工具、文件、接口或系统。
* **过程执行**：按照任务步骤持续推进，并根据执行结果调整后续动作。
* **上下文利用**：结合历史信息、用户偏好、任务背景和当前输入进行判断。
* **结果交付**：将处理结果整理成用户需要的形式，并发送到指定位置。

#### 3. 对比示例：会议纪要

* **对话型 AI 流程**：用户复制会议内容，AI 生成纪要，用户再复制回飞书或文档。
* **Agent 流程**：Agent 从飞书或指定入口接收任务，获取会议内容，整理会议结论、待办事项、负责人和截止时间，再将结果返回飞书或写入文档。
* **关键差异**：Agent 把一次文本生成扩展为“获取材料、整理结构、形成待办、交付结果”的完整流程。

#### 4. 对比示例：出差计划

* **对话型 AI 流程**：用户输入出差信息，AI 生成一份出差计划文本。
* **Agent 流程**：Agent 读取日程，查询交通和酒店信息，整理出差待办，生成计划，并发送到指定入口。
* **关键差异**：Agent 处理一组连续动作，适合需要多来源信息和多步骤执行的任务。

#### 5. 真实工作流中的连续任务

* **资料获取**：从文件、网页、消息、数据库或业务系统中找到相关资料。
* **信息判断**：判断资料价值、优先级、可信度和相关性。
* **结构整理**：将零散信息整理为大纲、清单、表格、报告或文档。
* **内容生成**：生成通知、纪要、总结、计划、脚本、报告或方案。
* **格式处理**：按照飞书文档、表格、消息卡片或其他格式要求进行整理。
* **结果发送**：将结果发送给指定对象、群组或系统。
* **归档跟进**：保存结果、记录状态，并在后续任务中继续使用。

### 三、OpenClaw 是什么

#### 1. OpenClaw 的出现背景

* **AI 需要进入工作现场**：真实工作发生在消息、文件、网页、表格、日程、脚本和业务系统之间。
* **个人 Agent 需要统一入口**：用户希望在熟悉的消息入口中调用 AI，并让 AI 处理个人环境中的任务。
* **创始人出发点**：Peter Steinberger 做 OpenClaw 的出发点，是希望 AI 能通过 WhatsApp、Telegram 等消息入口进入真实工作环境。
* **自托管助手需求增强**：个人和小团队希望拥有可控、可配置、可扩展的 AI 助手系统。

#### 2. OpenClaw 的基本定位

* **个人 AI 助手系统**：OpenClaw 可以运行在用户自己的电脑或服务器上，为个人提供持续可用的 AI 助手能力。
* **Agent Gateway**：OpenClaw 负责连接消息入口、模型能力、Agent、Skills 和自动化工具。
* **多入口助手**：OpenClaw 支持用户通过飞书、Telegram、Slack、Web UI 等入口使用 AI。
* **能力扩展平台**：OpenClaw 通过 Skills 和外部工具，把 AI 能力扩展到文件处理、消息处理、任务执行和自动化流程。

#### 3. OpenClaw 的直观理解

* **大模型是大脑**：负责理解、推理和生成。
* **OpenClaw 是调度系统**：负责接收消息、组织任务和连接能力。
* **消息入口是使用界面**：用户通过飞书等工具与 AI 对话。
* **Skills 是工具箱**：让 AI 具备更多可调用能力。
* **本地与外部工具是工作空间**：文件、脚本、API 和自动化任务共同构成 AI 的执行环境。

### 四、OpenClaw 为什么适合真实工作流

#### 1. 入口价值：在用户原本的地方使用 AI

* **飞书场景**：用户可以在飞书中提问、生成内容、整理会议纪要和处理团队协作信息。
* **即时沟通场景**：Telegram、Slack 等入口适合个人助手、团队通知和轻量任务处理。
* **Web UI 场景**：网页入口适合配置、调试、测试和管理助手能力。
* **入口价值**：用户可以在原本的工作界面中调用 AI，减少工具切换和材料搬运。

#### 2. 能力价值：把模型能力连接到文件、工具和系统

* **本地资料**：在配置相应权限和工具后，OpenClaw 可以围绕文件、目录、项目材料和个人知识进行处理。
* **外部服务**：在配置 API、脚本或自动化平台后，OpenClaw 可以连接网页服务、数据库和业务系统。
* **业务流程**：在配置相应 Skills 后，OpenClaw 可以参与资料整理、内容生成、消息推送和后续跟进。
* **能力扩展**：Skills 让不同用户可以根据自己的工作场景配置专属能力。

#### 3. 持续价值：把一次性问答沉淀为长期助手

* **任务连续性**：OpenClaw 适合围绕一个目标完成多步骤处理。
* **上下文积累**：在配置记忆和个人资料后，OpenClaw 可以结合个人偏好、历史资料和长期任务背景。
* **自动化处理**：在配置自动化任务后，OpenClaw 可以承担定时收集、自动整理、结果推送等重复流程。
* **工作流沉淀**：常见处理过程可以通过 Skills、规则和自动化任务沉淀为可复用能力。

### 五、OpenClaw 能用来干什么

#### 1. 消息与协作助手

* **即时问答**：在飞书等消息入口中直接向 OpenClaw 提问。
* **消息处理**：在飞书接入和必要权限配置后，整理群聊内容、总结会议记录、提炼项目进展和生成待办事项。
* **文本生成**：修改通知、公告、日报、周报、项目同步和会议提醒。
* **协作辅助**：将处理结果返回到团队沟通入口，辅助群公告、项目同步和协作文档生成。

#### 2. 资料整理与内容生成

* **多材料整理**：在提供资料或配置资料访问权限后，将多份材料整理成大纲、报告、清单或表格。
* **课程与培训材料**：根据已有文档生成培训讲稿、课程笔记和学习资料。
* **聊天记录提炼**：从聊天记录中提炼共识、分歧、行动项和负责人。
* **结构化输出**：将零散内容整理成适合传播、汇报或归档的结构化文档。

#### 3. 本地文件与个人知识助手

* **文件查找**：在配置本地权限后，定位本地文件和项目资料。
* **目录阅读**：围绕指定目录中的资料进行总结和问答。
* **项目整理**：辅助整理项目文件、生成说明文档和提取关键信息。
* **个人知识**：围绕学习、研究、课程、项目等主题维护资料积累，并根据历史资料回答问题。

#### 4. 自动化与长期助手

* **定时收集**：在配置相应工具后，收集行业新闻、网页信息、公告或指定信息源。
* **自动整理**：将收集到的资料整理成摘要、报告或清单。
* **自动推送**：在接入消息入口后，将整理结果推送到飞书或其他入口。
* **长期上下文**：在配置记忆、资料和规则后，逐渐贴近用户的工作习惯和任务需求。

## 第二部分：OpenClaw 的核心运行原理

### 一、OpenClaw 在系统里的位置

#### 1. 系统分层

* **入口层**：飞书、Web UI、Telegram、Slack。
* **调度层**：Gateway。
* **执行层**：Agent。
* **能力层**：Model、Tools、Skills、Plugins。
* **上下文层**：Workspace、Context、Memory。

#### 2. 运行关系

```text
用户
 ↓
Channel：飞书 / Web UI / Telegram / Slack
 ↓
Gateway：接收消息、管理会话、分发任务、返回结果
 ↓
Agent：理解任务、组织步骤、调用能力
 ↓
Model + Tools + Skills
 ↓
Workspace / Context / Memory
 ↓
结果返回原入口
```

#### 3. 核心角色

* **Channel**：外部消息入口。
* **Gateway**：中转和调度服务。
* **Agent**：任务执行者。
* **Model**：语言理解、推理、生成和工具选择能力。
* **Tools**：读文件、访问网页、执行脚本、发送消息等具体动作。
* **Skills**：完成某类任务的说明书和配套资源。
* **Plugins**：入口、工具、模型、Skills 等扩展能力包。
* **Workspace**：任务资料和文件所在的位置。
* **Context**：当前任务中模型能看到的信息。
* **Memory**：跨任务保留下来的长期信息。

---

### 二、一条消息如何被处理

#### 1. 消息处理链路

```text
用户发出任务
 ↓
Channel 接收消息
 ↓
Gateway 识别入口、用户、会话和权限
 ↓
Agent 准备上下文
 ↓
模型判断任务和处理方式
 ↓
Agent 调用 Tools / Skills
 ↓
模型整合中间结果
 ↓
Gateway 将结果返回原入口
```

#### 2. Channel

* 外部平台接入。
* 消息来源记录。
* 用户、群组、会话、入口类型识别。
* 结果回传所需信息保留。

#### 3. Gateway

* 多入口消息接收。
* 用户与会话识别。
* 权限与配置判断。
* Agent、模型、能力集合分发。
* 任务结果回传。

#### 4. Agent

* 会话历史读取。
* 系统提示、用户配置、Skills 加载。
* Workspace 资料读取。
* Memory 长期信息引入。
* 当前 Context 组织。

#### 5. Model 与 Tools

* 任务意图判断。
* 处理步骤规划。
* 工具调用选择。
* 工具返回结果整合。
* 最终内容生成。

---

### 三、Agent 的执行循环

#### 1. 模型判断

* 用户目标。
* 材料边界。
* 输出要求。
* 可用工具。
* 下一步动作。

#### 2. 工具调用

* 读取文件。
* 搜索网页。
* 访问 API。
* 执行脚本。
* 写入文档。
* 发送消息。

#### 3. 中间结果处理

* 工具返回内容。
* 错误信息。
* 权限状态。
* 文件变化。
* 外部系统反馈。

#### 4. 继续推进

* 结果检查。
* 步骤调整。
* 新工具调用。
* 输出整理。
* 任务交付。

---

### 四、Tools、Skills、Plugins 的关系

#### 1. Tool

* 具体动作。
* 由 Agent 调用。
* 返回可继续处理的结果。
* 典型例子：读文件、跑命令、查网页、发消息、写文档。

#### 2. Skill

* 某类任务的工作方法。
* 包含规则、流程、模板、参考资料和脚本。
* 影响 Agent 的处理顺序、检查标准和输出格式。
* 典型例子：会议纪要整理、课程讲稿改写、Slide 制作、表格分析。

#### 3. Plugin

* 一组扩展能力的安装包。
* 可以带来新的入口、工具、Skills、模型配置或 MCP 服务。
* 适合把一类能力整体接入 OpenClaw。

#### 4. 三者关系

```text
Plugin：安装一组能力
 ↓
Skill：规定任务做法
 ↓
Tool：执行具体动作
```

---

### 五、Workspace、Context、Memory

#### 1. Workspace

* 当前任务使用的文件空间。
* 课程文档、图片、PPT 源码、配置文件、生成结果。
* Agent 读取资料和生成文件的位置。

#### 2. Context

* 当前任务里模型能看到的信息。
* 用户输入、会话历史、系统提示、相关文件片段、Skill 内容、工具返回结果、当前时间和运行环境。
* 直接影响模型判断和输出质量。

#### 3. Memory

* 跨任务保留的信息。
* 用户偏好、常用规则、项目背景、历史事实。
* 长期助手持续贴近用户工作习惯的基础。

#### 4. 三者区别

```text
Workspace：资料和文件的位置
Context：当前任务正在使用的信息
Memory：后续任务继续使用的信息
```

#### 5. 课程资料整理例子

* **Workspace**：课程文档、图片、PPT 源码和生成文件。
* **Context**：当前请求、相关文件片段、对话历史和课程写作 Skill。
* **Memory**：讲稿口播偏好、Slide 学员视角、部署步骤详细要求。
* **Agent 执行**：读取资料、改写文案、生成结果、交付到指定文件。

## 第三部分：OpenClaw 的部署思路

### 一、部署方案总览

#### 1. 四种部署方式

* **本地部署**：运行在自己的电脑上。
* **云服务器部署**：运行在云服务器上。
* **第三方平台托管**：运行环境由平台提供。
* **Docker 部署**：运行在容器中。

#### 2. 选型维度

* 上手门槛。
* 控制权。
* 长期在线能力。
* 本地资料处理。
* 消息入口稳定性。
* 维护成本。
* 数据和权限边界。

---

### 二、本地部署

#### 1. 是什么

* OpenClaw 运行在自己的电脑上。
* macOS、Windows、Linux 都可以部署。
* 适合课堂跟做、个人学习、本地文件处理。

#### 2. 本地部署流程

* 系统环境检查。
* 安装脚本。
* `openclaw onboard`。
* 模型配置。
* Web UI / TUI 入口。
* 基础验证。
* 本地文件权限。

#### 3. 本地使用边界

* 电脑在线状态。
* 网络连接。
* 系统权限。
* 本地目录访问范围。
* 日志和配置文件位置。

---

### 三、云服务器部署

### 五、Docker 部署

#### 1. 是什么

* OpenClaw 运行在 Docker 容器中。
* 常见方式是 Docker Compose 管理服务、端口、环境变量和数据目录。
* 适合开发测试、隔离环境、多版本验证。

#### 2. 容器部署流程

* Docker 与 Docker Compose。
* 镜像版本。

* 环境变量。
