# 第二部分：OpenClaw 的核心运行原理

## 一、OpenClaw 在系统里的位置

### 1. 从外部看 OpenClaw

* **入口层**：用户通过飞书、Web UI、Telegram、Slack 等入口发起任务。
* **调度层**：Gateway 接收消息、管理会话、分发任务、返回结果。
* **执行层**：Agent 理解任务，组织步骤，调用模型和工具。
* **能力层**：模型、Tools、Skills、Plugins 提供具体处理能力。
* **上下文层**：Workspace、Context、Memory 提供任务材料、当前上下文和长期信息。

### 2. 总体结构

```text
用户
 ↓
Channel：飞书 / Web UI / Telegram / Slack
 ↓
Gateway：接收消息、管理会话、分发任务、返回结果
 ↓
Agent：理解任务、组织步骤、调用能力
 ↓
Model + Tools + Skills
 ↓
Workspace / Context / Memory
 ↓
结果返回原入口
```

### 3. 核心角色

* **Channel**：外部入口，把用户消息接入 OpenClaw。
* **Gateway**：中转和调度服务，负责连接、会话、分发和回传。
* **Agent**：任务执行者，把用户目标推进成一组处理动作。
* **Model**：提供语言理解、推理、生成和工具选择能力。
* **Tools**：Agent 可以实际调用的工具，例如读文件、访问网页、执行脚本、发送消息。
* **Skills**：教 Agent 完成某类任务的说明书和配套资源。
* **Workspace**：任务运行时使用的资料和文件空间。
* **Context**：当前这一轮任务中模型能看到的信息。
* **Memory**：跨任务保留下来的长期信息。

## 二、一条消息如何被处理

### 1. 基本链路

```text
用户发出任务
 ↓
Channel 接收消息
 ↓
Gateway 识别入口、用户、会话和权限
 ↓
Agent 准备上下文
 ↓
模型理解任务并判断处理方式
 ↓
Agent 调用 Tools / Skills
 ↓
模型整合中间结果
 ↓
Gateway 将结果返回原入口
```

### 2. Channel：把外部消息接进系统

* Channel 负责和外部平台打交道。
* 飞书 Channel 接收飞书机器人消息，Web UI Channel 接收网页端输入。
* Channel 会保留消息来源信息，例如用户、群组、会话、入口类型。
* 这些来源信息会影响后续会话归属、权限判断和结果回传。

### 3. Gateway：把消息变成可处理的任务

* Gateway 接收来自不同 Channel 的消息。
* Gateway 判断这条消息来自哪里、属于哪个用户、进入哪个会话。
* Gateway 根据配置决定使用哪个 Agent、模型和能力集合。
* Gateway 把任务交给 Agent，并在任务完成后负责结果回传。

### 4. Agent：准备处理任务所需的信息

* Agent 读取当前会话历史。
* Agent 加载系统提示、用户配置和可用 Skills。
* Agent 根据任务需要读取 Workspace 中的资料。
* Agent 结合 Memory 中的长期信息，形成当前任务的 Context。
* Context 准备完成后，模型开始判断任务意图和处理步骤。

### 5. 模型与工具调用

* 模型先判断用户要完成什么。
* 简单任务可以直接生成回复。
* 复杂任务会进入多步骤处理：读取资料、调用工具、检查结果、继续整理。
* Agent 按模型判断调用 Tools，工具返回结果后再交给模型继续处理。
* Skill 会影响任务执行方式，例如输出格式、检查步骤、需要参考的资料。

## 三、Agent 如何把模型能力变成任务执行

### 1. 模型提供判断能力

* **理解输入**：识别用户提问、任务目标、材料边界和隐含要求。
* **拆解任务**：把一个复杂请求拆成几个可执行环节。
* **选择工具**：判断当前任务需要文件、网页、命令、消息入口或其他能力。
* **整合结果**：把工具返回的信息整理成可交付内容。
* **生成输出**：生成回答、清单、表格、报告、代码或通知文本。

### 2. Agent 负责推进过程

* Agent 管理一次任务的执行流程。
* Agent 在模型判断和工具调用之间来回切换。
* Agent 根据中间结果继续推进后续步骤。
* Agent 把最终输出交还给 Gateway。

### 3. Tools、Skills、Plugins 的关系

```text
Plugin：安装一组扩展能力
 ↓
Skill：说明某类任务应该怎么完成
 ↓
Tool：执行具体动作
```

* **Tool** 是动作本身，例如读取文件、搜索网页、运行脚本、发送消息。
* **Skill** 是任务方法，例如“整理会议纪要时提取主题、结论、待办、负责人和时间”。
* **Plugin** 是能力安装包，可以带来新的入口、工具、模型或 Skills。

### 4. 一个任务执行例子

用户在飞书里让 OpenClaw 整理一段会议记录：

* Channel 把飞书消息送进 OpenClaw。
* Gateway 识别用户和会话，把任务交给 Agent。
* Agent 准备当前消息、历史上下文和会议纪要 Skill。
* 模型判断输出结构，提取会议主题、关键结论、待办事项、负责人和时间。
* 需要写入文档时，Agent 调用对应工具。
* 结果通过 Gateway 回到飞书。

这一套流程说明了 OpenClaw 的基本原理：模型负责判断，Agent 负责推进，工具负责行动，入口负责回到工作现场。

## 四、Workspace、Context、Memory 如何支撑长期任务

### 1. Workspace：任务材料所在的位置

* Workspace 是 OpenClaw 执行任务时使用的工作空间。
* 文件、配置、任务材料、生成结果都可以放在 Workspace 中。
* Agent 处理本地资料、项目文件、课程材料时，会围绕 Workspace 读取和生成内容。
* Workspace 像办公桌，资料放在这里，任务也在这里展开。

### 2. Context：当前任务能看到的信息

* Context 是当前这一轮任务里模型能使用的信息集合。
* Context 可能包含用户输入、会话历史、系统提示、相关文件片段、Skill 内容、工具返回结果、当前时间和运行环境。
* 模型的判断质量会受到 Context 影响。
* 同一个问题，Context 更完整，模型更容易给出贴合任务的处理结果。

### 3. Memory：跨任务保留的信息

* Memory 保存长期信息，例如用户偏好、常用规则、项目背景和历史事实。
* Memory 让 OpenClaw 在后续任务中继续利用以前积累的信息。
* 长期助手的价值来自持续积累：用户常做什么、偏好什么格式、哪些资料经常被使用。

### 4. 三者的区别

```text
Workspace：任务资料和文件所在的位置
Context：当前任务中模型正在使用的信息
Memory：后续任务还能继续使用的长期信息
```

### 5. 课程资料整理例子

学员让 OpenClaw 整理课程资料时：

* Workspace 里放着课程文档、图片、PPT 源码和生成文件。
* Context 里包含当前请求、相关文件片段、对话历史和课程写作 Skill。
* Memory 里记录长期偏好，例如“讲稿要适合口播”“Slide 要面向学员”“部署步骤要详细”。
* Agent 结合这三类信息完成资料整理、文案改写和结果生成。

这就是 OpenClaw 从一次性问答走向长期工作流的基础：入口让任务进入现场，Gateway 管理链路，Agent 推进过程，模型做判断，工具执行动作，上下文和记忆保存工作材料。
