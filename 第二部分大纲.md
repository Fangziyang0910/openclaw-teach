# 第二部分：理解 OpenClaw 的基本原理、核心架构和运行逻辑

## 一、从用户视角进入系统视角

### 1. 用户看到的 OpenClaw

* **一个可对话的 AI 助手**：用户可以通过飞书、Telegram、Slack、Web UI 等入口与 OpenClaw 对话。
* **一个能处理任务的助手**：OpenClaw 可以接收用户请求，结合模型和工具完成内容生成、资料整理、消息处理和自动化任务。
* **一个可以长期运行的系统**：OpenClaw 可以部署在本地电脑或服务器上，持续为用户提供 AI 助手能力。

### 2. 系统内部的 OpenClaw

* **消息入口层**：负责接收来自飞书、Telegram、Slack、Web UI 等入口的消息。
* **Gateway 调度层**：负责统一接收消息、管理连接、分发任务和返回结果。
* **Agent 执行层**：负责理解任务、组织步骤、调用模型和工具。
* **能力扩展层**：通过模型、Tools、Skills 和 Plugins 扩展具体能力。
* **工作空间层**：通过 Workspace、Context 和 Memory 维护任务上下文和长期资料。

### 3. 第二部分的核心理解方式

* **入口负责接收请求**。
* **Gateway 负责统一调度**。
* **Agent 负责推进任务**。
* **模型负责理解和生成**。
* **Tools 和 Skills 负责扩展能力**。
* **Workspace 和 Memory 负责保存上下文与长期信息**。

## 二、OpenClaw 的整体架构

### 1. 总体结构

```text
用户
 ↓
消息入口 Channel
飞书 / Telegram / Slack / Web UI / 其他入口
 ↓
OpenClaw Gateway
 ↓
Agent
 ↓
模型 + Tools + Skills
 ↓
文件 / API / 脚本 / 自动化任务 / Memory
 ↓
结果返回原入口
```

### 2. 架构中的关键角色

* **Channel**：用户与 OpenClaw 对话的入口。
* **Gateway**：OpenClaw 的调度中心。
* **Agent**：任务执行者，负责理解任务并推进流程。
* **Model**：智能能力来源，负责理解、推理和生成。
* **Tools**：Agent 可以直接调用的具体工具。
* **Skills**：告诉 Agent 如何完成某类任务的能力说明。
* **Workspace**：Agent 执行任务时使用的工作空间。
* **Memory**：保存长期信息、偏好和历史资料的机制。

### 3. OpenClaw 的基本运行思想

* 用户从熟悉的入口发出任务。
* Gateway 接收并分发任务。
* Agent 结合上下文理解任务。
* 模型判断任务意图和处理方式。
* Tools 和 Skills 提供具体执行能力。
* 结果通过原入口返回给用户。

---

## 三、Gateway：OpenClaw 的调度中心

### 1. Gateway 的基本作用

* **接收消息**：接收来自飞书、Telegram、Slack、Web UI 等入口的用户请求。
* **管理连接**：维护消息入口、Web UI、命令行工具和其他客户端之间的连接。
* **分发任务**：将用户请求交给合适的 Agent 和模型处理。
* **协调工具**：在任务需要时协调 Tools、Skills、文件、脚本和外部服务。
* **返回结果**：将 Agent 的处理结果发送回用户原来的入口。

### 2. Gateway 的通俗理解

* Gateway 可以理解为 OpenClaw 的“总调度台”。
* 用户消息先进入 Gateway。
* Gateway 决定这条消息属于哪个会话、应该交给哪个 Agent、是否需要调用工具。
* Agent 完成任务后，结果再通过 Gateway 返回用户。

### 3. Gateway 与后续实操的关系

* 部署 OpenClaw 时，需要确认 Gateway 是否正常启动。
* 接入飞书时，飞书机器人需要把消息送到 Gateway。
* 配置 Skills 时，Gateway 需要知道当前系统可以使用哪些能力。
* 排查问题时，Gateway 状态是最重要的检查点之一。

---

## 四、Channel：用户与 OpenClaw 的入口

### 1. Channel 的基本概念

* Channel 是 OpenClaw 连接外部消息平台的入口。
* 用户通过 Channel 给 OpenClaw 发消息。
* OpenClaw 通过 Channel 把结果返回给用户。
* 不同 Channel 对应不同使用场景。

### 2. 常见 Channel 类型

* **飞书**：适合国内团队协作、课堂实操和办公场景。
* **Telegram**：适合个人助手和轻量消息任务。
* **Slack**：适合国际化团队协作。
* **Web UI**：适合调试、配置和本地测试。
* **其他入口**：根据社区生态和插件能力扩展到更多平台。

### 3. Channel 的主要价值

* **降低使用门槛**：用户可以在熟悉的消息工具中使用 AI。
* **贴近工作现场**：很多任务本来就产生于群聊、私聊、项目沟通和团队协作中。
* **支持结果回流**：AI 处理结果可以直接回到原来的消息入口。
* **便于团队协作**：同一个助手可以服务于个人对话、群聊讨论和协作流程。

### 4. 飞书 Channel 的课堂定位

* 飞书是本课程的主要实操入口。
* 学员将通过飞书机器人与 OpenClaw 对话。
* 飞书接入展示的是 OpenClaw 如何从本地系统进入真实办公入口。
* 飞书场景可以自然承接会议纪要、群聊总结、日报周报、任务整理等案例。

---

## 五、Agent：任务执行者

### 1. Agent 的基本作用

* **理解任务**：判断用户真正想完成什么。
* **组织步骤**：将任务拆成几个可执行环节。
* **调用能力**：根据任务需要调用模型、Tools、Skills 或外部服务。
* **处理结果**：把中间结果整理成最终回复。
* **交付输出**：把结果返回到飞书、Web UI 或其他入口。

### 2. Agent 的运行方式

* 用户发来请求。
* Agent 读取当前会话和相关上下文。
* Agent 判断是否需要调用工具。
* Agent 结合工具结果继续处理任务。
* Agent 生成最终结果并交给 Gateway 返回。

### 3. Agent 与普通对话 AI 的区别

* 普通对话 AI 主要生成回答。
* Agent 需要围绕目标推进任务。
* 普通对话 AI 更像“回答者”。
* Agent 更像“任务处理者”。

### 4. Agent 在 OpenClaw 中的位置

* Gateway 负责接收和分发任务。
* Agent 负责真正处理任务。
* 模型为 Agent 提供理解和生成能力。
* Tools 和 Skills 为 Agent 提供外部执行能力。
* Workspace 和 Memory 为 Agent 提供上下文支持。

---

## 六、Model：智能能力来源

### 1. 模型的基本作用

* **语言理解**：理解用户输入的自然语言。
* **任务判断**：判断用户想要完成的目标。
* **推理规划**：决定任务应该如何处理。
* **内容生成**：生成回答、总结、文案、计划、代码或报告。
* **工具选择**：在需要时判断是否调用工具或 Skills。

### 2. OpenClaw 与模型的关系

* OpenClaw 本身负责连接和调度。
* 模型负责提供智能能力。
* 同一个 OpenClaw 系统可以接入不同模型。
* 模型能力会影响任务理解、工具选择和结果质量。

### 3. 常见模型来源

* OpenAI
* Claude
* Gemini
* DeepSeek
* Kimi
* 通义千问 / Qwen
* 其他 OpenAI-compatible API
* 本地模型服务

### 4. 课堂中的理解方式

* **模型是大脑**：负责理解、推理和生成。
* **OpenClaw 是系统**：负责把大脑连接到入口、工具和工作流。
* **Agent 是执行者**：负责把模型能力转化为任务处理过程。

---

## 七、Tools、Skills 与 Plugins

### 1. Tools：Agent 可以直接调用的工具

* **文件工具**：读取、写入、编辑文件。
* **命令工具**：执行命令行或脚本。
* **网页工具**：搜索网页、读取网页信息。
* **消息工具**：发送消息或处理消息内容。
* **媒体工具**：处理图片、语音、视频等内容。
* **自动化工具**：执行定时任务或流程任务。

### 2. Skills：面向任务的能力说明

* Skill 可以理解为“能力说明书”。
* Skill 告诉 Agent：

  * 这个能力适合什么任务。
  * 使用这个能力需要哪些输入。
  * 执行任务时应该遵循什么步骤。
  * 输出结果应该是什么形式。
* Skills 让 OpenClaw 可以适配更多具体场景。

### 3. Plugins：能力扩展包

* Plugin 可以为 OpenClaw 增加新的入口、工具、模型或 Skills。
* Plugin 更像一组能力的安装包。
* 通过 Plugins，OpenClaw 可以持续扩展生态能力。

### 4. 三者的关系

```text
Plugin：扩展能力来源
 ↓
Skill：说明如何使用能力
 ↓
Tool：Agent 实际调用的具体工具
```

### 5. 课堂中的理解方式

* **Tool 是工具本身**。
* **Skill 是使用工具的方法说明**。
* **Plugin 是安装和扩展能力的方式**。
* 对普通用户来说，配置 Skills 是让 OpenClaw 变得更有用的关键步骤。

---

## 八、Workspace、Context 与 Memory

### 1. Workspace：Agent 的工作空间

* Workspace 是 OpenClaw 执行任务时使用的工作目录。
* 本地文件、配置文件、任务材料和部分上下文都可以放在 Workspace 中。
* Agent 在处理任务时，会围绕 Workspace 中的资料进行读取、整理和生成。
* Workspace 可以理解为 OpenClaw 的“工作台”。

### 2. Context：当前任务的上下文

* Context 是模型在当前任务中能看到的信息。
* Context 可能包括：

  * 用户当前输入
  * 当前对话历史
  * 系统提示词
  * 相关文件内容
  * Skills 说明
  * 工具调用结果
  * 当前时间和运行环境信息
* Context 决定模型本轮回答时掌握了哪些材料。

### 3. Memory：长期记忆

* Memory 用于保存长期信息。
* Memory 可以记录用户偏好、长期事实、常用规则和历史资料。
* Memory 让 OpenClaw 更适合长期助手场景。
* 长期助手的价值来自持续积累，而不仅是单次回答。

### 4. 三者的区别

```text
Workspace：资料和配置所在的工作空间
Context：当前任务中模型正在使用的信息
Memory：系统长期保存、后续还能继续使用的信息
```

### 5. 课堂中的直观理解

* Workspace 像办公桌。
* Context 像本次任务摊开的材料。
* Memory 像长期笔记本。
* Agent 每次处理任务时，会在办公桌上使用当前材料，并可能把重要信息记入长期笔记本。

## 九、一次消息的完整运行逻辑

### 1. 用户发送任务

* 用户在飞书、Web UI 或其他入口发送消息。
* 消息可能是一句提问、一段材料、一个文件、一个指令或一个任务目标。
* 这个消息首先进入对应的 Channel。

### 2. Channel 接收消息

* Channel 将外部平台中的消息转交给 OpenClaw。
* Channel 会保留必要的来源信息，例如用户、会话、群组或入口类型。
* Channel 将消息交给 Gateway 处理。

### 3. Gateway 进行调度

* Gateway 接收消息。
* Gateway 判断消息来自哪个入口、哪个用户、哪个会话。
* Gateway 根据配置和权限判断是否处理这条消息。
* Gateway 将任务交给 Agent。

### 4. Agent 准备上下文

* Agent 读取当前会话历史。
* Agent 加载必要的系统提示和配置。
* Agent 读取相关 Workspace 信息。
* Agent 识别可用 Tools 和 Skills。
* Agent 结合 Memory 中的长期信息。

### 5. 模型理解与规划

* 模型理解用户任务。
* 模型判断任务类型。
* 模型决定直接回答，或进一步调用工具。
* 对简单任务，模型可以直接生成结果。
* 对复杂任务，模型会进入多步骤处理流程。

### 6. Tools 和 Skills 执行任务

* Agent 根据模型判断调用合适工具。
* 工具可能读取文件、搜索信息、执行脚本、处理资料或发送消息。
* Skill 为任务提供执行方法和格式要求。
* 工具返回结果后，模型继续整理和判断。

### 7. 生成最终结果

* Agent 将中间结果整理成最终输出。
* 模型生成适合用户阅读的回复。
* Agent 可以按照任务要求输出文本、清单、表格、报告或通知内容。

### 8. 结果返回原入口

* Gateway 接收 Agent 输出。
* Gateway 将结果交给对应 Channel。
* Channel 把结果发送回飞书、Web UI 或其他入口。
* 用户在原来的工作位置看到结果。

### 9. 任务记录与长期积累

* 会话记录可以被保存。
* 重要上下文可以进入 Memory。
* 任务中的文件和结果可以保存在 Workspace。
* 后续任务可以继续利用已有资料和历史信息。

## 十、以飞书为例理解运行链路

### 1. 飞书私聊场景

```text
用户在飞书私聊机器人
 ↓
飞书 Channel 接收消息
 ↓
Gateway 判断用户和会话
 ↓
Agent 读取上下文
 ↓
模型理解任务
 ↓
必要时调用 Tools / Skills
 ↓
生成回复
 ↓
结果返回飞书私聊
```

### 2. 飞书群聊场景

```text
用户在群里 @机器人
 ↓
飞书 Channel 接收群聊消息
 ↓
Gateway 判断群组策略和用户权限
 ↓
Agent 处理任务
 ↓
模型生成结果或调用工具
 ↓
结果返回群聊
```

### 3. 飞书任务示例：整理会议纪要

* 用户在飞书中发送会议内容或让机器人整理讨论。
* OpenClaw 接收消息并进入对应会话。
* Agent 判断任务是“会议纪要整理”。
* 模型提取会议主题、关键结论、待办事项、负责人和时间。
* 必要时调用飞书相关能力生成文档或返回群聊。
* 用户在飞书中得到结构化会议纪要。

### 4. 飞书任务示例：生成项目同步

* 用户在飞书中输入项目进展材料。
* Agent 识别需要生成团队同步内容。
* 模型按照项目背景和表达要求生成结构化文本。
* OpenClaw 将结果返回飞书。
* 用户可以直接复制、修改或发送给团队。

## 十一、权限、安全与边界

### 1. 为什么需要权限控制

* OpenClaw 可以连接消息入口、文件、工具、API 和脚本。
* 这些能力越强，越需要明确边界。
* 系统需要知道谁可以使用助手，助手可以做什么，哪些操作需要谨慎。

### 2. 用户访问边界

* **私聊权限**：控制哪些用户可以在私聊中使用机器人。
* **群聊权限**：控制机器人是否响应群聊消息。
* **@ 触发机制**：群聊中通过 @机器人 降低误触发风险。
* **白名单机制**：只允许指定用户或群组使用。

### 3. 工具能力边界

* **文件权限**：控制 Agent 可以读取或修改哪些文件。
* **命令权限**：控制 Agent 是否可以执行命令行或脚本。
* **网络权限**：控制 Agent 是否可以访问网页或外部 API。
* **消息权限**：控制 Agent 是否可以发送消息、更新文档或操作协作平台。
* **自动化权限**：控制 Agent 是否可以执行定时任务或长期任务。

### 4. Skills 和 Plugins 的安全边界

* Skills 会影响 Agent 如何执行任务。
* Plugins 可能带来新的入口、工具和系统能力。
* 第三方 Skills 和 Plugins 需要关注来源、权限和具体功能。
* 涉及文件操作、命令执行、外部 API、消息发送的能力需要特别谨慎。

### 5. 课堂中的安全原则

* 先从私聊场景开始。
* 先配置最小权限。
* 先用测试任务验证。
* 涉及删除、发送、修改、执行脚本等操作时保留人工确认。
* 将 OpenClaw 当作具备实际行动能力的助手，而不是单纯聊天机器人。
