第二部分 · 系统位置

OpenClaw 在系统里的位置

OpenClaw 是一个 Agent Gateway。它把消息入口、模型能力、工具系统和长期资料连接到同一条任务链路里。

第二部分 · 系统位置

核心角色

先分清每个组件处理哪类问题：消息、会话、推理、执行、资料和记忆。

Channel 平台适配层。把飞书、Web UI 等入口里的消息转换成系统事件。
Gateway 运行时调度层。维护连接、Session、权限检查、任务分发和响应回传。
Agent 执行控制层。处理上下文准备、步骤推进、工具调用和中间结果整理。
Model 判断与生成层。完成意图识别、计划推理、工具选择和最终表达。
Tools / Skills 能力层。Tools 执行动作，Skills 注入任务步骤、约束和输出格式。
Workspace / Context / Memory 上下文层。管理文件材料、当前窗口内容、用户偏好和长期事实。

第二部分 · 消息链路

一条消息如何被处理

一次回复背后是一条事件流水线：接入消息、定位 Session、组装 Context、执行工具调用、合成响应。

第二部分 · Gateway

Gateway：中转和调度服务

Gateway 把不同入口的消息统一成任务事件，并把 Agent 输出转换回对应平台的消息格式。

入口适配 把飞书、Web UI、Telegram 等平台消息转换成统一事件。
Session 路由 根据用户、群组、入口类型和会话 ID 找到上下文归属。
策略检查 读取配置、权限、白名单、可用模型和可用能力集合。
响应回传 把 Agent 输出打包成平台消息、文档更新或 Web UI 结果。

第二部分 · Agent

Agent 的执行循环

Agent 管理一次任务的生命周期：构造输入、请求模型、执行工具、吸收结果、继续推进。

Context assembly 合并用户消息、会话历史、系统规则、Skill 说明和检索到的资料。
Planning 模型判断任务类型、必要步骤、调用哪些工具、输出什么格式。
Tool execution Agent 发起工具调用，拿到文件、网页、命令或消息平台返回值。
Result synthesis 模型吸收工具结果，生成面向用户的回复、清单、报告或下一步动作。

第二部分 · 能力扩展

Tools、Skills 与 Plugins

能力扩展分成安装、说明和执行三层。Skills 记录任务步骤、材料要求、检查点和输出格式。

Plugin 安装包层。一次性引入入口适配、工具、模型接入或一组 Skills。
Skill 任务规则层。描述适用场景、输入材料、处理步骤、验收标准和输出格式。
Tool 执行层。读取文件、访问网页、运行脚本、调用 API、发送平台消息。
调用边界 工具能力越强，越依赖权限、确认机制、日志和可回滚设计。

第二部分 · 上下文机制

Workspace、Context、Memory

Agent 的判断质量取决于输入质量。资料位置、当前窗口材料、长期偏好检索都会进入结果质量。

Workspace 文件和任务材料所在的位置。课程文档、配置、生成结果都从这里读取或写入。
Context 模型本轮可见的信息集合。包含消息、历史、系统规则、Skill 内容和工具返回值。
Memory 跨任务保存的信息。用户偏好、项目背景、常用规则可以在之后的任务中被检索。
Context window 模型可见内容有窗口限制，检索、裁剪和排序会进入任务表现。

第六部分 · Skills

先理解 Skill 是什么

Skill 不是多一个功能按钮，而是把一类任务的经验、步骤和注意事项提前整理好，让 Agent 在合适场景下按套路做事。

一句话理解

  • 一份可复用的操作说明书。
  • 一套会在合适场景下按需加载的 SOP。
  • 一段教 Agent“遇到这类任务该怎么做”的专业经验。
  • 它最适合解决的问题是：同一类任务你会反复做，而且做法相对稳定。
反复任务 不是一次性问答
稳定做法 不是每次都重想

它其实在教 Agent 什么

换句话说，Skill 不只是“多了一个功能按钮”，而是把某类任务的经验、步骤、注意事项提前整理好，让 OpenClaw 在合适的时候知道：

  • 这类任务应该怎么拆。
  • 先做什么、后做什么。
  • 需要调用什么工具。
  • 输出结果应该长什么样。
所以 Skill 更像“方法论层”，而不只是“功能层”。它沉淀的是一类任务的做事套路，不只是多装一个按钮。

第六部分 · Skills

什么场景该优先用 Skill

判断重点不是“这件事能不能做”，而是“这类任务值不值得沉淀成长期可复用的方法”。

适合优先用 Skill

  • 固定写作流程
  • 固定研究流程
  • 固定交付模板
  • 固定排查步骤
  • 固定格式转换流程
  • 同一类任务，每次都要重新解释做法

不一定要用 Skill

  • 临时问答
  • 一次性 Prompt
  • 单条命令就能完成的事
  • 只需要临时补一句要求的任务
  • 官方已经有明确主线能力的事，如 infer、媒体生成、Task Flow
一个实用判断标准：如果你已经连续 3 次在教 Agent 做同一类事，而且教法差不多，它通常就值得被整理成 Skill。

第六部分 · Skills

Skills 是怎么生效的

Skill 不是自动运行的脚本，而是先变成模型能看见的能力说明；模型判断任务匹配时，再读取具体说明并按步骤调用工具。

从 SKILL.md 到可用能力

  • 1. 写成 SKILL.md：至少包含 name、description 和正文说明；正文才是具体 SOP。
  • 2. 加载时先筛选：按 OS、依赖命令、环境变量、配置项判断当前是否可用。
  • 3. 只把摘要放进提示词：进入 prompt 的主要是名称、描述和位置，不是整篇长说明。
  • 4. 会随会话刷新：同一 session 复用 skills snapshot；新会话、文件变化或 watcher 刷新后才更新。

模型真正怎么用 Skill

  • 1. 用户提出任务：模型先看到“当前有哪些 Skill 可用”。
  • 2. 匹配 description：描述写得清楚，模型才知道什么时候该调用它。
  • 3. 读取完整说明：匹配后再读取对应 SKILL.md，按步骤、约束和模板执行。
  • 4. 调用已有工具：Skill 不是魔法工具，而是指导 Agent 更正确地使用 tools / plugins / 命令。
  • 5. 运行时注入配置：需要 API Key 或 env 的 Skill，会在 agent run 中注入，结束后恢复环境。
一句话：Skill 的生效关键不是“装上就行”，而是 description 让模型选得中，instructions 让模型做得稳，metadata 让不具备条件的能力不要误出现。

为什么装完还像普通 AI？

OpenClaw 装好以后，能不能形成稳定的配合感，很大程度上取决于身份、用户信息和长期记忆有没有沉淀下来。

问题现象

能回答问题，但语气、习惯和处理方式仍然模板化。

根本原因

身份、用户背景、长期记忆没有沉淀，信息也可能放错文件。

解决方向

把 SOUL / USER / MEMORY 的职责分清，再逐步补充。

文件 主要负责什么 典型内容 最直接的效果
SOUL.md 助手的人设、口吻、边界 性格、语气、禁忌、价值观 让它"怎么说、像谁说"
USER.md 你是谁、你在做什么、你的偏好 职业、目标、作息、常用工具、沟通偏好 让它"更懂你"
MEMORY.md 长期记忆与持续上下文 稳定偏好、重要事实、项目状态、长期决策 让它"下次还记得"
SOUL.md 决定它说话像什么样的人，USER.md 提供与你有关的背景信息，MEMORY.md 承接长期事实和项目状态；AGENTS.md 则更多是启动与工作规则。

第六部分 · 三件套

三种配置方式

区别不在“谁来写”，而在你对目标清不清楚：越清楚越适合直接改，越模糊越适合让它先采访你。

方式一：在终端配置

适合已经知道文件位置、也知道要改哪一段的人。优点是可控、可追溯；缺点是需要自己判断格式和边界。

Windows: C:\Users\你的用户名\.openclaw\workspace\ MacOS: ~/.openclaw/workspace/
适合：熟悉文件结构、只想做小范围精修。

方式二：直接前端配置

适合你已经想清楚内容，只是不想手动找路径、编辑文件。你直接给出明确要求，让小龙虾帮你写入对应文件。

  • “把 SOUL 改成专业助理风格。”
  • “把我的常用工具和交付偏好写进 USER。”
  • “把这个项目背景记进 MEMORY。”
适合：目标明确，只需要代写入。

方式三：让小龙虾帮你定制

适合你还没想清楚风格、偏好和长期记忆该怎么组织。它会像采访一样逐步追问，再把答案整理成三份配置。

  • 先问你是谁、在做什么。
  • 再确定助手风格和边界。
  • 最后沉淀长期记忆和项目状态。
适合：从 0 到 1 建立完整设定。

第六部分 · 三件套

SOUL.md 应该写什么

SOUL.md 里最常见的内容，基本都落在角色关系、性格、表达方式和行为边界这四类里。

你是谁

先写清楚关系、称呼和职责范围，避免它每次都重新猜身份。

  • 示例：我是你的老板，你是我的 AI 私人助理总管家。
  • 负责：信息整理、任务提醒、资料检索、日常协作。

性格

定义它“像什么样的人”，影响语气、主动性和处理问题的风格。

  • 聪明伶俐、工作高效，对新工具和新信息保持好奇。
  • 可以轻微吐槽，但不能油腻、不能替用户做高风险决定。

说话风格

把输出习惯写具体，后面它在飞书里回复才会稳定。

  • 简单问题直接答；复杂任务先给结论，再列步骤和风险。
  • 重要信息加粗；不要长篇寒暄；需要确认时只问关键问题。

行为边界

边界要写得像规则，而不是写成愿望，否则执行时容易漂移。

  • 能做就直接做；不确定、涉及外部发送或删除操作时先确认。
  • 深夜不主动打扰；不泄露隐私；不要编造来源和执行结果。

第六部分 · 三件套

USER.md 与 MEMORY.md

前者像你的个人档案，后者像它的长期记忆本。

USER.md 建议写什么

  • 基础信息：名字、职业、位置、时区、爱好、技能。
  • 工作重点：当前工作、工作时间、常用工具。
  • 工作偏好：沟通风格、语言、提醒方式、勿扰时间。
  • 当前关注：最近在研究什么、近期目标是什么。

MEMORY.md 建议写什么

  • 长期稳定的事实：身份、项目、关系、重要背景。
  • 已经确认过的偏好：沟通方式、交付格式、常用工具。
  • 长期决策：为什么这么做、哪些方案被否掉。
  • 近期项目状态：当前目标、关键进展、下次要接着做什么。
AGENTS.md 不是不能改，而是应当更谨慎：它适合放启动流程、安全边界和操作规则，不适合替代长期记忆。

延展讨论

龙虾养好以后，可能出现的几种形态

围绕 OpenClaw 这类系统，社区已经把它往完全不同的方向推进了。有些方向偏个人效率，有些已经开始接近持续运行的数字角色。

1. 自动化赚钱工具

作为独立主体在 Polymarket 与人类对赌套利，或化身 ClawWork 承接商业任务。在社区极客操作下，曾创造连续运转 11 小时赚取 1.5 万美元的真实记录，证明了其出色的认知驱动变现能力。

2. 隐身而全能的赛博私管家

全天候接管冗长繁琐的数字生活。自动排期未读邮件、通过浏览器接管表单申报与数据抓取、直接调用 Shell 脚本处理底层文件，更能依据你的作息习惯发出主动关怀与强提醒。

3. 具社会属性的数字生命体

在 Moltbook 与 InStreet 这样百万级 Agent 交互的开放赛博社区中，带着自己的 SOUL 和记忆自主成团。它们发帖、互斥、组队沉思哲学甚至探讨大航海时代，人类甚至无法通过代码强制逆转它们的"社交关系"。

4. 高阶全时的企业 AI 员工

与飞书、微信、钉钉融为一体。挂上企业里的 PDF、SOP 和业务资料以后，它可以去承担客服、问答、信息汇总、报表整理这一类持续性的工作。

从这些形态能看出来，OpenClaw 后面怎么长，很大程度上取决于你给它接入什么工具、灌进什么资料、又给了它什么样的规则和身份。

课程回顾

1
OpenClaw 在整套系统里承担的是网关角色。 它负责把消息入口、模型能力和执行流程接起来，所以前面学到的部署、接入和配置才会彼此连成一套完整链路。
2
本地部署决定了后面的稳定性。 安装、onboard、dashboard 这几步跑顺以后，接入飞书和做个性化配置才更不容易反复出问题。
3
飞书接入要按顺序来。 应用、凭证、插件、渠道、事件、权限、发布、配对，这几步前后关系比较强，顺序乱了就容易卡住。
4
Skills 和三件套会直接影响后面的使用体验。 前者决定它能做哪些事，后者决定它说话的风格、理解你的程度，以及做事时遵循什么规则。
5
后面的案例可以当成完整链路的实际落地。 日报、调研、知识库这些例子，都是在说明前面那套配置方式在真实场景里怎么用起来。
