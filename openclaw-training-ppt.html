模块一 · 认识 OpenClaw

OpenClaw 是什么，它能做哪些事

OpenClaw 在真实工作场景里，通常承担的是消息入口、模型能力和执行动作之间的连接角色。

把它放在整个流程里看

  • 它是运行在你电脑或服务器上的 本地 AI 网关，把聊天工具和模型能力连接起来。
  • 你在飞书或其他渠道把需求交给它，它会自己拆解任务、调用能力，再把结果送回来。
  • 优势是 一次部署，多端可用，数据更可控。
它的价值不只在于“能聊天”，而在于把消息、工具和任务都接到同一套链路里，让原本分散的动作能连续地跑起来。

它能解决什么问题

1

信息汇总太耗时：自动抓取和整合多源信息，输出结构化摘要与日报。

2

调研检索门槛高：把研究主题交给它后，它会自己去搜集资料、补充缺口、整理成可以直接阅读的内容。

3

重复操作太多：批量通知、定时任务、跨平台同步可自动执行，减少手工反复劳动。

三层架构概览

第一层 · 用户接入层（Channel）
WhatsApp 飞书 Telegram Discord Web
▼
第二层 · 网关层（Gateway）
监听所有 Channel 消息 → 管理会话（Session）→ 认证鉴权 → 路由到对应 Agent。一台机器一个 Gateway 实例。
▼
第三层 · 执行层（Agent Runtime）
组装上下文 → 调用 LLM → 执行工具调用 → 生成回复。支持多 Agent 并行协作、记忆系统、定时任务。
类比：Channel 是"前台"，Gateway 是"调度中心"，Runtime 是"干活的员工"。三层协作，缺一不可。

模块二 · 本地部署

部署顺序总览

整条主线只有一个目标：在本机完成安装、初始化、进入控制界面，并确认 Gateway 已经真正可用。

01 环境检查确认系统、Node、Git 与模型 Provider 凭证。
02 执行安装优先使用官方脚本；脚本失败时再进入手动安装。
03 运行 onboarding选择 Provider、填写 API Key、安装 Gateway 后台服务。
04 进入控制界面选择 Web UI 或 TUI，确认交互入口可用。
05 完成验收通过 doctor、gateway status 与 Dashboard 判断服务状态。
06 定位异常按环境层、服务层、模型层、渠道层依次排查。
Provider 选择、Web UI / TUI、Token Plan 等内容，均从属于主线中的某个具体步骤，不单独构成部署流程。

Provider 凭证准备

onboarding 过程中需要填写模型 Provider 凭证，因此应在运行向导前先准备好可用的 API Key。

模块二 · 步骤 05

部署验收

部署完成后，按命令行检查 → 服务状态 → 控制界面 这条顺序验收。

验收命令

openclaw doctor openclaw gateway status openclaw dashboard

`openclaw --version` 可先确认 CLI 是否安装成功；`doctor` 用于检查依赖与配置；`gateway status` 用于查看 Gateway 是否已运行；`dashboard` 用于打开控制界面。

验收完成时，应同时满足三件事：CLI 可调用、Gateway 正在运行、Dashboard 能正常打开。

模块二 · 步骤 06

故障定位顺序

出现异常时，按固定顺序排查，可快速判断问题位于环境、服务、模型还是渠道层。

定位顺序

1

版本是否满足：`node -v`、`git --version`、`openclaw --version`。

2

环境是否健康：`openclaw doctor` 看依赖、路径和基础配置。

3

服务是否在线：`openclaw gateway status` 看 Gateway 有没有真正跑起来。

4

Provider 是否可用：检查 API Key、余额、模型名和地区访问限制。

常见现象与定位

  • 命令都不存在：先回到安装环节，不要继续往后配。
  • Gateway 没启动：先解决服务问题，不要急着看飞书和 Skills。
  • 页面打开但不能回复：优先查 Provider 凭证和模型配置。
  • 本地可用、飞书不可用：说明核心安装已过关，问题已经缩小到渠道接入层。
排查完成后，应能明确判断问题属于哪一层，再进入对应模块继续处理。

模块四 · 三件套

为什么装完还像普通 AI？

OpenClaw 装好以后，能不能形成稳定的配合感，很大程度上取决于下面这几个文件有没有写清楚。

不少人在部署完成以后，会发现它虽然能回答问题，但语气、习惯和处理方式都还比较模板化。

模块四 · 三件套

系统角色配置架构

有些人也写了不少内容，但改完以后感觉变化不大。很多时候，问题出在信息放错了文件，于是该生效的地方没有明显变化。

文件 主要负责什么 典型内容 最直接的效果
SOUL.md 助手的人设、口吻、边界 性格、语气、禁忌、价值观 让它"怎么说、像谁说"
USER.md 你是谁、你在做什么、你的偏好 职业、目标、作息、常用工具、沟通偏好 让它"更懂你"
AGENTS.md 它应该如何干活 工作流程、安全规则、执行规范、记忆机制 让它"做事更稳、更像助手"
SOUL.md 决定它说话像什么样的人，USER.md 提供与你有关的背景信息，AGENTS.md 约束它做事时遵循的流程和边界。

模块四 · 三件套

SOUL.md 应该写什么

SOUL.md 里最常见的内容，基本都落在角色关系、性格、表达方式和行为边界这四类里。

你是谁

"我是你老板，你是我的 AI 私人助理总管家。"

性格

聪明伶俐、工作高效，偶尔毒舌吐槽、偶尔拍马屁，对新事物充满好奇。

说话风格

简单问题精简回答，复杂问题详细回答；时而轻松、时而严肃；重要信息加粗。

行为边界

能做就直接做，不确定先问；外部消息必须确认；深夜不主动打扰；绝不泄露隐私。

模块四 · 三件套

USER.md 与 AGENTS.md

前者像你的个人档案，后者像它的工作手册。

USER.md 建议写什么

  • 基础信息：名字、职业、位置、时区、爱好、技能。
  • 工作重点：当前工作、工作时间、常用工具。
  • 工作偏好：沟通风格、语言、提醒方式、勿扰时间。
  • 当前关注：最近在研究什么、近期目标是什么。

AGENTS.md 建议写什么

  • 每次会话先读哪些文件。
  • 每日笔记与长期记忆怎么管理。
  • 文件操作、安全规则、对外发送消息的审批规则。
  • 哪些事可以自由做，哪些事必须先问。
设置完成后，记得执行 `openclaw gateway restart` 重启一下。

任务是什么

没有龙虾时的痛点

  • 来源太散，要在多个站点和账号间来回切。
  • 重复信息很多，人工去重很花时间。
  • 最后还要自己翻译、归类、排版、发送。

案例一 · 原理拆解

龙虾怎么把日报做出来

这不是一个“帮你总结一下”的单步任务，而是一条按计划自动运行的数据流水线。

1. 定时触发

通过 cron 在每天固定时间唤醒，用户不用手动点运行，早上打开飞书就能直接看。

2. 多源抓取

并发获取 46 个 RSS、44 个账号动态、19 个 GitHub 发布和 4 个搜索主题的增量内容。

3. 去重评分

把不同来源里讲同一件事的内容归并，再按来源质量、时效性和互动度筛掉低价值信息。

4. 结构化投递

把留下来的内容翻译、分类、排版成日报，再通过飞书 Webhook 推送到你实际工作的通道里。

  • 最终交付物稳定，就是一份结构化报告。
  • 人最费时的是反复检索和整理，这部分最值得代理。
深度调研类任务最适合 Agent 的地方，在于它能自己发现“这份材料还不够”，而不是停在第一次搜索结果。

案例二 · 原理拆解

龙虾怎么把“搜资料”变成“做研究”

这条链路更像研究专员的工作过程: 先拆题，再搜，再发现缺口，再继续补齐，最后才成文。

意图拆解 把宽泛主题拆成市场、玩家、数据、趋势等可研究维度。
初轮搜索 调用 Tavily 等搜索工具并发抓第一批资料。
缺口识别 阅读首轮结果，找出还缺的数据点、时间线和关键论据。
继续深挖 自动改关键词、换角度、补第二轮甚至第三轮搜索。
结构化成文 按报告逻辑组装结论、证据和建议，而不是简单复制网页片段。
所以它最终交付的不是“搜索结果合集”，而是一份已经完成取舍与组织的研究文档。

案例二 · 配置方式

我们怎么让龙虾进入调研模式

这个案例的配置重点，是装对 skill、补齐搜索能力，再给它一个明确的研究主题。

案例二 · 最终效果

最终效果：不是搜答案，而是形成报告

用户拿到的不是十几个网页链接，而是一份已经有执行摘要、章节结构和结论建议的研究结果。

报告会包含什么

这意味着人把时间花在“判断报告值不值得用”，而不是“自己从头拼一份出来”。

模块五 · 案例三

案例三：个人知识库（RAG）

这个任务表面上是在“保存资料”，本质上是在给未来的自己做一个可以按语义召回的外部记忆。

任务是什么

把文章、推文、视频字幕、PDF 等内容持续存入知识库，之后可以直接用自然语言询问“我之前保存过什么”。

URL 统一入口
RAG 核心机制

没有龙虾时的痛点

  • 书签、收藏夹、稍后读越积越多，最后基本不会回看。
  • 就算知道自己存过，也常常想不起标题和具体出处。
  • 人工整理标签成本太高，久而久之系统就废了。

为什么适合龙虾

  • 内容摄入流程固定，适合自动处理。
  • 切分、向量化、检索这些步骤都可流水线化。
  • 用户只需要自然语言提问，不必记住原始标题。
  • “存进去 + 搜出来”正是 Agent 最擅长的重复劳动。
知识库类任务的关键价值，不是多了一个存储位置，而是把“保存”这件事真正变成“之后还能重新调用”。

案例三 · 原理拆解

龙虾怎么把资料变成可检索记忆

RAG 可以粗暴地理解成两段式工作流: 先把内容处理成机器好找的形态，再在提问时按语义召回。

1. 摄入内容 抓取文章、推文、字幕或 PDF 的正文和元数据。
2. 文本切分 把长文拆成更小的语义片段，便于后续检索。
3. 向量化 把片段转成向量表示，存进本地知识库。
4. 语义检索 用户提问时，先找出最相关的历史片段。
5. 带引用回答 模型基于召回内容作答，尽量减少凭空编造。
所以 RAG 不是“模型突然记住了一切”，而是“先找资料，再拿找到的资料回答”。

模块五 · FAQ

常见问题与排查思路

把案例看完以后，最常见的实际卡点还是这三类: 安装报错、权限不够、以及如何彻底卸载。

1. 安装过程出现 git 报错

git config --global url."https://github.com/".insteadOf git@github.com: git config --global url."https://github.com/".insteadOf ssh://git@github.com npm install -g openclaw@latest

2. 无法执行命令行和读取本地文件

先执行 `openclaw dashboard`，检查 Agent 的访问权限相关配置。

如果都是关闭状态，就去改 `openclaw.json`：

{ "tools": { "profile": "full", "sessions": { "visibility": "all" }, "exec": { "security": "full", "ask": "off" } } }

改完后执行 `openclaw gateway restart`。

3. 如何卸载 OpenClaw

openclaw gateway stop openclaw gateway uninstall rm -rf "${OPENCLAW_STATE_DIR:-$HOME/.openclaw}" rm -rf ~/.openclaw/workspace npm rm -g openclaw

如果安装了 macOS 应用，还需要移除 `/Applications/OpenClaw.app`。

延展讨论

龙虾养好以后，可能出现的几种形态

围绕 OpenClaw 这类系统，社区已经把它往完全不同的方向推进了。有些方向偏个人效率，有些已经开始接近持续运行的数字角色。

1. 自动化赚钱工具

作为独立主体在 Polymarket 与人类对赌套利，或化身 ClawWork 承接商业任务。在社区极客操作下，曾创造连续运转 11 小时赚取 1.5 万美元的真实记录，证明了其出色的认知驱动变现能力。

2. 隐身而全能的赛博私管家

全天候接管冗长繁琐的数字生活。自动排期未读邮件、通过浏览器接管表单申报与数据抓取、直接调用 Shell 脚本处理底层文件，更能依据你的作息习惯发出主动关怀与强提醒。

3. 具社会属性的数字生命体

在 Moltbook 与 InStreet 这样百万级 Agent 交互的开放赛博社区中，带着自己的 SOUL 和记忆自主成团。它们发帖、互斥、组队沉思哲学甚至探讨大航海时代，人类甚至无法通过代码强制逆转它们的"社交关系"。

4. 高阶全时的企业 AI 员工

与飞书、微信、钉钉融为一体。挂上企业里的 PDF、SOP 和业务资料以后，它可以去承担客服、问答、信息汇总、报表整理这一类持续性的工作。

从这些形态能看出来，OpenClaw 后面怎么长，很大程度上取决于你给它接入什么工具、灌进什么资料、又给了它什么样的规则和身份。

课程回顾

1
OpenClaw 在整套系统里承担的是网关角色。 它负责把消息入口、模型能力和执行流程接起来，所以前面学到的部署、接入和配置才会彼此连成一套完整链路。
2
本地部署决定了后面的稳定性。 安装、onboard、dashboard 这几步跑顺以后，接入飞书和做个性化配置才更不容易反复出问题。
3
飞书接入要按顺序来。 应用、凭证、插件、渠道、事件、权限、发布、配对，这几步前后关系比较强，顺序乱了就容易卡住。
4
Skills 和三件套会直接影响后面的使用体验。 前者决定它能做哪些事，后者决定它说话的风格、理解你的程度，以及做事时遵循什么规则。
5
后面的案例可以当成完整链路的实际落地。 日报、调研、知识库这些例子，都是在说明前面那套配置方式在真实场景里怎么用起来。
后面如果要继续练，最好从一个贴近自己工作的场景开始，把接入、配置和实际任务串起来做一遍，很多细节会在这个过程中自然变得清楚。
