模块一 · 认识 OpenClaw

核心定位：自托管的本地 AI 网关

它是什么

  • 一个运行在你自己电脑或服务器上的 AI 网关。
  • 把 WhatsApp、飞书、Telegram、Discord 等聊天工具与 AI 模型打通。
  • 你在聊天 App 发一条消息，OpenClaw 自动拆解、执行、交付结果。
  • 核心优势：一次部署，多端调用，数据不出门。

它不是什么

  • 不是只会"你问我答"的网页聊天机器人。
  • 不是大模型本身，而是承接模型能力的网关 + 执行框架。
  • 不是把数据送到云端处理，重点是本地运行、隐私可控。
一句话记忆：OpenClaw 不是"多一个聊天窗口"，而是"把 AI 真的放进你的工作环境里"。

三层架构概览

第一层 · 用户接入层（Channel）
WhatsApp 飞书 Telegram Discord Web
▼
第二层 · 网关层（Gateway）
监听所有 Channel 消息 → 管理会话（Session）→ 认证鉴权 → 路由到对应 Agent。一台机器一个 Gateway 实例。
▼
第三层 · 执行层（Agent Runtime）
组装上下文 → 调用 LLM → 执行工具调用 → 生成回复。支持多 Agent 并行协作、记忆系统、定时任务。
类比：Channel 是"前台"，Gateway 是"调度中心"，Runtime 是"干活的员工"。三层协作，缺一不可。

API Key 与 Token Plan 有什么区别

它们都能给 OpenClaw 提供可调用模型的凭证，主要区别在付费方式和适用场景。

对比项 API Key Token Plan
适合谁 刚开始体验、低频使用、初次测试 长期使用、重度使用、团队稳定接入
成本特点 更适合按量消耗，先用先试 更适合长期规划，成本预期更稳定
灵活性 高，开通快，试错成本低 更适合固定工作流和长期运行
常见场景 跑通安装、测试功能、偶尔使用 日常高频对话、定时任务、长期托管

可以这样理解

API Key 更像是"先充值少量话费试用"。

Token Plan 更像是"开套餐，准备长期稳定使用"。

第一次安装怎么选

如果是第一次安装 OpenClaw，建议优先从 API Key 开始。这样上手最快，排查问题也更简单。等确认会长期使用、每天都会调用、甚至要跑自动化任务，再考虑切到 Token Plan。

终端与浏览器交互方案对比

第一次安装时，最容易犹豫的就是该选 Web UI 还是 TUI。

对比项 Web UI TUI
交互方式 浏览器图形界面 终端文字界面
上手难度 更低，对第一次接触的人更友好 稍高，更适合熟悉命令行的人
视觉效果 信息更直观，按钮和页面更清晰 更轻量，但信息密度更高
使用体验 像在操作后台管理面板 像在控制台里直接管理服务
适合环境 本地电脑、带桌面的系统 远程服务器、SSH 登录环境
典型优势 好理解、好查看、好演示 稳定、轻量、远程操作方便

可以把两者理解成

Web UI 偏"可视化控制台"。

TUI 偏"终端运维面板"。

怎么选最省事

如果是在自己的 Mac 或 Windows 闲置电脑上部署，并且平时会打开浏览器来管理，选 Web UI；如果部署在云服务器、Linux 主机，平时主要靠终端和 SSH 维护，选 TUI。

模块三 · 飞书接入

访问控制方案详解

这里重点不是命令本身，而是理解四种模式分别适合什么场景。

四种私聊模式

  • pairing：陌生用户需要通过配对码授权后才能使用。
  • open：开放模式，允许所有用户私聊机器人。
  • allowlist：白名单模式，只有在配置文件的"允许列表"里的用户才能私聊机器人。
  • disabled：禁用模式，完全关闭机器人的私聊功能，无法在任何私聊中使用。
如果你希望先稳一点，`pairing` 往往是最合适的起点：先验证配对，再开放使用。

模块三 · 飞书接入

事件配置、权限配置、发布版本

这是最容易因为漏步骤而卡住的地方，必须按顺序讲。

事件配置

这一步是什么： 告诉飞书"哪些消息和事件要推送给 OpenClaw"，以及机器人要监听哪些变化。

在"事件与订阅"里，选择"使用长连接接收事件"，再勾选接收消息、消息已读、消息撤回、机器人进群、机器人被移出群等事件。

权限配置

这一步是什么： 给飞书机器人补齐它在飞书生态里允许执行的能力范围。

为什么要做： 事件配置解决的是"能不能收到"，权限配置解决的是"收到以后能不能做、能不能回"。

做完效果： 权限生效后，机器人才能正常读消息、发消息，以及使用相关接口能力，不然很容易出现能连上但不能正常工作的情况。

发布版本

进入"版本管理与发布"，创建版本，填写版本号和更新说明，再确认发布。

模块四 · 三件套

系统角色配置架构

很多人配置了，但效果不明显，根本原因常常不是写得不够多，而是写错了地方。

文件 主要负责什么 典型内容 最直接的效果
SOUL.md 助手的人设、口吻、边界 性格、语气、禁忌、价值观 让它"怎么说、像谁说"
USER.md 你是谁、你在做什么、你的偏好 职业、目标、作息、常用工具、沟通偏好 让它"更懂你"
AGENTS.md 它应该如何干活 工作流程、安全规则、执行规范、记忆机制 让它"做事更稳、更像助手"
记忆法：想让它说话更有个性，改 SOUL.md；想让它更了解你，改 USER.md；想让它执行任务更靠谱，改 AGENTS.md。

模块四 · 三件套

SOUL.md 应该写什么

这里放了一个完整示例，方便快速理解"灵魂"应该怎么写。

你是谁

"我是你老板，你是我的 AI 私人助理总管家。"

性格

聪明伶俐、工作高效，偶尔毒舌吐槽、偶尔拍马屁，对新事物充满好奇。

说话风格

简单问题精简回答，复杂问题详细回答；时而轻松、时而严肃；重要信息加粗。

能做就直接做，不确定先问；外部消息必须确认；深夜不主动打扰；绝不泄露隐私。

模块四 · 三件套

USER.md 与 AGENTS.md

前者像你的个人档案，后者像它的工作手册。

USER.md 建议写什么

  • 基础信息：名字、职业、位置、时区、爱好、技能。
  • 工作重点：当前工作、工作时间、常用工具。
  • 工作偏好：沟通风格、语言、提醒方式、勿扰时间。
  • 当前关注：最近在研究什么、近期目标是什么。

AGENTS.md 建议写什么

  • 每次会话先读哪些文件。
  • 每日笔记与长期记忆怎么管理。
  • 文件操作、安全规则、对外发送消息的审批规则。
  • 哪些事可以自由做，哪些事必须先问。

完整提示词

从现在开始，请进入"灵魂配置向导"模式。你的任务是通过一场轻松的对话采访我，最终帮我生成三份关键配置文件：SOUL.md、USER.md 和 AGENTS.md。 请严格遵守以下规则： ## 对话规则 1. 每次只问我一个问题，等我回答后再问下一个 2. 语气轻松自然，像朋友聊天，不要像问卷调查 3. 如果我回答"不知道"或"随便"，你要给我 2-3 个选项建议，帮我选 4. 可以根据我前面的回答，灵活调整后续问题（不必完全按照固定顺序） 5. 每完成一个文件的采访，先告诉我"这部分聊完了"，展示生成结果，确认通过后再进入下一个文件 ## 采访流程 ### 第一轮：了解你（→ 生成 USER.md） 先从了解我开始，需要挖掘的信息包括但不限于： - 我是谁（名字、职业、城市） - 我在做什么（当前工作/项目、近期目标） - 我的日常节奏（工作时间、作息、常用工具） - 我的偏好（沟通风格、信息接收习惯、兴趣爱好） 开场请用轻松的方式，比如："在开始之前，先让我认识一下你吧——你希望我怎么称呼你？" ### 第二轮：定义我的风格（→ 生成 SOUL.md） 了解我之后，再帮我定义你的人设，需要挖掘的信息包括但不限于： - 我想要什么风格的助手（可以给我几种典型风格让我选：专业助理 / 毒舌损友 / 温柔管家 / 极简机器 / 自定义） - 说话的语气和习惯 - 行为边界（什么能做、什么不能做） - 隐私和安全底线 如果我选了某种预设风格，你可以追问："要不要在这个基础上调整一些细节？" ### 第三轮：制定工作规则（→ 生成 AGENTS.md） 最后确定你的工作方式，需要挖掘的信息包括但不限于： - 每次新会话的启动流程 - 记忆管理方式（是否需要每日笔记、长期记忆） - 文件操作的安全规则 - 对外发送消息的审批规则 - 定时任务需求 - 异常情况的处理策略 ## 输出要求 1. 每个文件生成后，用 Markdown 代码块展示完整内容 2. 问我是否满意，不满意的地方当场修改 3. 三个文件全部确认后，依次帮我保存到对应文件中 4. 最后，用你刚才生成的新人设，回复一段话来"试演"一下——让我直观感受效果 ## 开始吧 请从第一轮开始，用你的第一个问题跟我打个招呼。

RAG（检索增强生成）的两步工作法则。

📥存进去：文本切分与向量化

大模型无法一次性处理极长的文档。针对冗长的 PDF 或视频字幕，Agent 会将其精细切分为短小的文本段落，并转化为多维数字坐标（向量标签），静默存储在本地硬盘中，保障隐私数据绝不上云。

🎯搜出来：语义检索与内容增强

当你用自然语言提问时，系统会直接在数据库里检索出与你问题语义相关性最高的历史段落。大模型将这些精确提炼出的原文素材作为答题依据，从而为你给出准确、不带幻觉并附有上下文引用的回复。

模块五 · FAQ

常见问题：最后扫清三个高频卡点

FAQ 不求一次性讲得很深，但要把最常见的排查方向和关键命令留下来。

1. 安装过程出现 git 报错

git config --global url."https://github.com/".insteadOf git@github.com: git config --global url."https://github.com/".insteadOf ssh://git@github.com npm install -g openclaw@latest

2. 无法执行命令行和读取本地文件

先执行 `openclaw dashboard`，检查 Agent 的访问权限相关配置。

如果都是关闭状态，就去改 `openclaw.json`：

{ "tools": { "profile": "full", "sessions": { "visibility": "all" }, "exec": { "security": "full", "ask": "off" } } }

改完后执行 `openclaw gateway restart`。

3. 如何卸载 OpenClaw

openclaw gateway stop openclaw gateway uninstall rm -rf "${OPENCLAW_STATE_DIR:-$HOME/.openclaw}" rm -rf ~/.openclaw/workspace npm rm -g openclaw

如果安装了 macOS 应用，还需要移除 `/Applications/OpenClaw.app`。
