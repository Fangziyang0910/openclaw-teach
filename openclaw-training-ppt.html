模块四 · 本地部署

部署顺序总览

整条主线只有一个目标：在本机完成安装、初始化、进入控制界面，并确认 Gateway 已经真正可用。

01 环境检查确认系统、Node、Git 与模型 Provider 凭证。
02 执行安装优先使用官方脚本；脚本失败时再进入手动安装。
03 运行 onboarding选择 Provider、填写 API Key、安装 Gateway 后台服务。
04 进入控制界面选择 Web UI 或 TUI，确认交互入口可用。
05 完成验收通过 doctor、gateway status 与 Dashboard 判断服务状态。
06 定位异常按环境层、服务层、模型层、渠道层依次排查。
Provider 选择、Web UI / TUI、Token Plan 等内容，均从属于主线中的某个具体步骤，不单独构成部署流程。

执行安装命令

OpenClaw 官方推荐先使用安装脚本。脚本会完成 CLI 安装，并进入 onboarding 流程。

macOS / Linux

curl -fsSL https://openclaw.ai/install.sh | bash

在终端中直接执行上面的命令。

Windows（PowerShell）

Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned iwr -useb https://openclaw.ai/install.ps1 | iex

先放开当前用户脚本权限，再执行安装命令。Windows 原生可用，但更推荐 WSL2。

模块四 · 步骤 05

部署验收

部署完成后，按命令行检查 → 服务状态 → 控制界面 这条顺序验收。

验收命令

openclaw doctor openclaw gateway status openclaw dashboard

`openclaw --version` 可先确认 CLI 是否安装成功；`doctor` 用于检查依赖与配置；`gateway status` 用于查看 Gateway 是否已运行；`dashboard` 用于打开控制界面。

验收完成时，应同时满足三件事：CLI 可调用、Gateway 正在运行、Dashboard 能正常打开。

模块四 · 步骤 06

故障定位顺序

出现异常时，按固定顺序排查，可快速判断问题位于环境、服务、模型还是渠道层。

定位顺序

1

版本是否满足：`node -v`、`git --version`、`openclaw --version`。

2

环境是否健康：`openclaw doctor` 看依赖、路径和基础配置。

3

服务是否在线：`openclaw gateway status` 看 Gateway 有没有真正跑起来。

4

Provider 是否可用：检查 API Key、余额、模型名和地区访问限制。

常见现象与定位

  • 命令都不存在：先回到安装环节，不要继续往后配。
  • Gateway 没启动：先解决服务问题，不要急着看飞书和 Skills。
  • 页面打开但不能回复：优先查 Provider 凭证和模型配置。
  • 本地可用、飞书不可用：说明核心安装已过关，问题已经缩小到渠道接入层。
排查完成后，应能明确判断问题属于哪一层，再进入对应模块继续处理。

添加机器人能力

从本地走向多端驻留

打通这一环，系统会实质性脱离浏览器限制，在即时通讯软件中开启二十四小时待命状态。

两端配置对齐

大部分失败案例源于飞书后台配置遗漏。必须确保两端的事件订阅与发布信息完全匹配，服务才能接收动作消息。

渠道中枢映射

系统通过渠道设置实现身份互认，从而确认依靠哪个底层身份往飞书环境发送结果。

模块五 · 飞书接入

创建应用、获取凭证、添加机器人

飞书后台这三步分别对应应用主体、调用凭证和机器人能力，少了任何一项，后面的接入都会卡住。

飞书后台要完成的三项准备

Step 1

进入飞书开发者后台，创建企业自建应用。

Step 2

在"凭据与基础信息"里保存 AppID 和 AppSecret。

Step 3

在"添加应用能力"里把机器人能力加上。

应用创建完成以后，OpenClaw 侧才能拿着 AppID、AppSecret 和机器人能力去完成后续绑定。

访问控制方案详解

这四种模式决定的是谁能说话、谁能触发任务、谁能拿到结果；不同团队和群聊环境，适合的模式也不一样。

四种私聊模式

  • pairing：陌生用户需要通过配对码授权后才能使用。
  • open：开放模式，允许所有用户私聊机器人。
  • allowlist：白名单模式，只有在配置文件的"允许列表"里的用户才能私聊机器人。
  • disabled：禁用模式，完全关闭机器人的私聊功能，无法在任何私聊中使用。
`pairing` 模式会更稳一些：先完成配对，再逐步开放使用范围，排查问题也更容易。

模块六 · 三件套

为什么装完还像普通 AI？

OpenClaw 装好以后，能不能形成稳定的配合感，很大程度上取决于下面这几个文件有没有写清楚。

不少人在部署完成以后，会发现它虽然能回答问题，但语气、习惯和处理方式都还比较模板化。

实际上，小龙虾是需要养的，而且怎么养很重要。除了各式各样的 Skills，有三个配置文件非常重要。

三个文件的分工

SOUL定义性格、口吻、边界
USER定义你是谁、偏好什么
AGENTS定义它怎么干活

模块六 · 三件套

系统角色配置架构

有些人也写了不少内容，但改完以后感觉变化不大。很多时候，问题出在信息放错了文件，于是该生效的地方没有明显变化。

文件 主要负责什么 典型内容 最直接的效果
SOUL.md 助手的人设、口吻、边界 性格、语气、禁忌、价值观 让它"怎么说、像谁说"
USER.md 你是谁、你在做什么、你的偏好 职业、目标、作息、常用工具、沟通偏好 让它"更懂你"
AGENTS.md 它应该如何干活 工作流程、安全规则、执行规范、记忆机制 让它"做事更稳、更像助手"
SOUL.md 决定它说话像什么样的人，USER.md 提供与你有关的背景信息，AGENTS.md 约束它做事时遵循的流程和边界。

模块六 · 三件套

SOUL.md 应该写什么

SOUL.md 里最常见的内容，基本都落在角色关系、性格、表达方式和行为边界这四类里。

你是谁

"我是你老板，你是我的 AI 私人助理总管家。"

性格

聪明伶俐、工作高效，偶尔毒舌吐槽、偶尔拍马屁，对新事物充满好奇。

说话风格

简单问题精简回答，复杂问题详细回答；时而轻松、时而严肃；重要信息加粗。

行为边界

能做就直接做，不确定先问；外部消息必须确认；深夜不主动打扰；绝不泄露隐私。

模块六 · 三件套

USER.md 与 AGENTS.md

前者像你的个人档案，后者像它的工作手册。

USER.md 建议写什么

  • 基础信息：名字、职业、位置、时区、爱好、技能。
  • 工作重点：当前工作、工作时间、常用工具。
  • 工作偏好：沟通风格、语言、提醒方式、勿扰时间。
  • 当前关注：最近在研究什么、近期目标是什么。

AGENTS.md 建议写什么

  • 每次会话先读哪些文件。
  • 每日笔记与长期记忆怎么管理。
  • 文件操作、安全规则、对外发送消息的审批规则。
  • 哪些事可以自由做，哪些事必须先问。
设置完成后，记得执行 `openclaw gateway restart` 重启一下。

模块六 · 三件套

通过灵魂向导定制化配置

如果是第一次配这三份文件，用这种方式会轻松一些。它会一边问，一边把内容慢慢整理出来，不需要你一开始就把所有设定想得很完整。

向导的规则

  • 每次只问一个问题。
  • 先了解你，再定义助手风格，最后制定工作规则。
  • 每完成一个文件，先展示生成结果，再进入下一个。
  • 三个文件都确认后，再保存并"试演"一段新人设效果。

模块六 · 三件套

灵魂配置向导完整提示词

这段提示词会把原本要自己慢慢整理的三份文件，改造成一段循序渐进的问答过程。

完整提示词

从现在开始，请进入"灵魂配置向导"模式。你的任务是通过一场轻松的对话采访我，最终帮我生成三份关键配置文件：SOUL.md、USER.md 和 AGENTS.md。 请严格遵守以下规则： ## 对话规则 1. 每次只问我一个问题，等我回答后再问下一个 2. 语气轻松自然，像朋友聊天，不要像问卷调查 3. 如果我回答"不知道"或"随便"，你要给我 2-3 个选项建议，帮我选 4. 可以根据我前面的回答，灵活调整后续问题（不必完全按照固定顺序） 5. 每完成一个文件的采访，先告诉我"这部分聊完了"，展示生成结果，确认通过后再进入下一个文件 ## 采访流程 ### 第一轮：了解你（→ 生成 USER.md） 先从了解我开始，需要挖掘的信息包括但不限于： - 我是谁（名字、职业、城市） - 我在做什么（当前工作/项目、近期目标） - 我的日常节奏（工作时间、作息、常用工具） - 我的偏好（沟通风格、信息接收习惯、兴趣爱好） 开场请用轻松的方式，比如："在开始之前，先让我认识一下你吧——你希望我怎么称呼你？" ### 第二轮：定义我的风格（→ 生成 SOUL.md） 了解我之后，再帮我定义你的人设，需要挖掘的信息包括但不限于： - 我想要什么风格的助手（可以给我几种典型风格让我选：专业助理 / 毒舌损友 / 温柔管家 / 极简机器 / 自定义） - 说话的语气和习惯 - 行为边界（什么能做、什么不能做） - 隐私和安全底线 如果我选了某种预设风格，你可以追问："要不要在这个基础上调整一些细节？" ### 第三轮：制定工作规则（→ 生成 AGENTS.md） 最后确定你的工作方式，需要挖掘的信息包括但不限于： - 每次新会话的启动流程 - 记忆管理方式（是否需要每日笔记、长期记忆） - 文件操作的安全规则 - 对外发送消息的审批规则 - 定时任务需求 - 异常情况的处理策略 ## 输出要求 1. 每个文件生成后，用 Markdown 代码块展示完整内容 2. 问我是否满意，不满意的地方当场修改 3. 三个文件全部确认后，依次帮我保存到对应文件中 4. 最后，用你刚才生成的新人设，回复一段话来"试演"一下——让我直观感受效果 ## 开始吧 请从第一轮开始，用你的第一个问题跟我打个招呼。

RAG原理拆解

RAG（检索增强生成）的两步工作法则。

📥存进去：文本切分与向量化

大模型无法一次性处理极长的文档。针对冗长的 PDF 或视频字幕，Agent 会将其精细切分为短小的文本段落，并转化为多维数字坐标（向量标签），静默存储在本地硬盘中，保障隐私数据绝不上云。

延展讨论

龙虾养好以后，可能出现的几种形态

围绕 OpenClaw 这类系统，社区已经把它往完全不同的方向推进了。有些方向偏个人效率，有些已经开始接近持续运行的数字角色。

1. 自动化赚钱工具

作为独立主体在 Polymarket 与人类对赌套利，或化身 ClawWork 承接商业任务。在社区极客操作下，曾创造连续运转 11 小时赚取 1.5 万美元的真实记录，证明了其出色的认知驱动变现能力。

全天候接管冗长繁琐的数字生活。自动排期未读邮件、通过浏览器接管表单申报与数据抓取、直接调用 Shell 脚本处理底层文件，更能依据你的作息习惯发出主动关怀与强提醒。

3. 具社会属性的数字生命体

在 Moltbook 与 InStreet 这样百万级 Agent 交互的开放赛博社区中，带着自己的 SOUL 和记忆自主成团。它们发帖、互斥、组队沉思哲学甚至探讨大航海时代，人类甚至无法通过代码强制逆转它们的"社交关系"。

4. 高阶全时的企业 AI 员工

与飞书、微信、钉钉融为一体。挂上企业里的 PDF、SOP 和业务资料以后，它可以去承担客服、问答、信息汇总、报表整理这一类持续性的工作。

从这些形态能看出来，OpenClaw 后面怎么长，很大程度上取决于你给它接入什么工具、灌进什么资料、又给了它什么样的规则和身份。
