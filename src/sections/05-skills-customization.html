模块四 · Skills

先理解 Skill 是什么

Skill 不是多一个功能按钮，而是把一类任务的经验、步骤和注意事项提前整理好，让 Agent 在合适场景下按套路做事。

一句话理解

  • 一份可复用的操作说明书。
  • 一套会在合适场景下按需加载的 SOP。
  • 一段教 Agent“遇到这类任务该怎么做”的专业经验。
  • 它最适合解决的问题是：同一类任务你会反复做，而且做法相对稳定。
反复任务 不是一次性问答
稳定做法 不是每次都重想

它其实在教 Agent 什么

换句话说，Skill 不只是“多了一个功能按钮”，而是把某类任务的经验、步骤、注意事项提前整理好，让 OpenClaw 在合适的时候知道：

  • 这类任务应该怎么拆。
  • 先做什么、后做什么。
  • 需要调用什么工具。
  • 输出结果应该长什么样。
所以 Skill 更像“方法论层”，而不只是“功能层”。它沉淀的是一类任务的做事套路，不只是多装一个按钮。

什么场景该优先用 Skill

判断重点不是“这件事能不能做”，而是“这类任务值不值得沉淀成长期可复用的方法”。

适合优先用 Skill

  • 固定写作流程
  • 固定研究流程
  • 固定交付模板
  • 固定排查步骤
  • 固定格式转换流程
  • 同一类任务，每次都要重新解释做法

不一定要用 Skill

  • 临时问答
  • 一次性 Prompt
  • 单条命令就能完成的事
  • 只需要临时补一句要求的任务
  • 官方已经有明确主线能力的事，如 infer、媒体生成、Task Flow
一个实用判断标准：如果你已经连续 3 次在教 Agent 做同一类事，而且教法差不多，它通常就值得被整理成 Skill。

模块四 · Skills

Skills 是怎么生效的

想真正用好 Skill，不能只会“装一个就完事”，还得知道它的来源、优先级、加载顺序，以及为什么它会直接影响 Agent 的表现。

三层来源与优先级

OpenClaw 的 Skill 有三个来源，优先级从高到低排列。理解这一层，才知道为什么同名 Skill 会“覆盖生效”。

  • workspace：项目级，只影响当前工作区，优先级最高。
  • user：用户级，全局生效，通常来自 ClawHub 或手动安装。
  • bundled：内置 Skills，随版本发布，开箱即用，优先级最低。
  • 同名 Skill 会按优先级覆盖，因此可以在项目里重写内置行为。

真正的生效链路

当 OpenClaw 启动或收到任务时，Skills 通常按这条链路工作：

  1. 读取各层级的 SKILL.md，解析描述、触发条件和依赖。
  2. 检查环境变量，缺少必要配置的 Skill 会被跳过。
  3. 把可用 Skill 描述注入 system prompt，让模型知道自己能做什么。
  4. 任务结束后恢复上下文和环境状态，避免互相干扰。

为什么这会直接影响效果

  • 写得好：Agent 更像“知道套路的老手”。
  • 写得差：容易乱用工具、输出不稳定，甚至把任务做偏。
一句话：Skill 不是装饰品，而是 OpenClaw 组织能力和稳定输出的方式。

模块四 · Skills

社区 Skills 库：ClawHub

Skills 会直接影响 OpenClaw 的能力边界。

ClawHub 是做什么的

ClawHub 管的是 Skills 的查找、安装和更新，很多扩展能力都是从这里进入 OpenClaw 的。

npm i -g clawhub 安装 ClawHub 命令行工具，让本机具备搜索和安装 Skills 的入口。
clawhub search "postgres" 按任务关键词搜索可用 Skill，先看有没有现成方案。
clawhub install skill-name 把选中的 Skill 安装到本地，后续 OpenClaw 才能加载。
clawhub update --all 统一更新已安装 Skills；不再需要时再执行卸载命令移除。

模块四 · 三件套

为什么装完还像普通 AI？

OpenClaw 装好以后，能不能形成稳定的配合感，很大程度上取决于身份、用户信息和长期记忆有没有沉淀下来。

不少人在部署完成以后，会发现它虽然能回答问题，但语气、习惯和处理方式都还比较模板化。

实际上，小龙虾是需要养的。除了各式各样的 Skills，SOUL / USER / MEMORY 这三类配置也很重要。

三个文件的分工

SOUL定义性格、口吻、边界
USER定义你是谁、偏好什么
MEMORY沉淀长期事实与项目状态

模块四 · 三件套

系统角色配置架构

有些人也写了不少内容，但改完以后感觉变化不大。很多时候，问题出在信息放错了文件：该放长期记忆的内容，被误写进了工作规则。

文件 主要负责什么 典型内容 最直接的效果
SOUL.md 助手的人设、口吻、边界 性格、语气、禁忌、价值观 让它"怎么说、像谁说"
USER.md 你是谁、你在做什么、你的偏好 职业、目标、作息、常用工具、沟通偏好 让它"更懂你"
MEMORY.md 长期记忆与持续上下文 稳定偏好、重要事实、项目状态、长期决策 让它"下次还记得"
SOUL.md 决定它说话像什么样的人，USER.md 提供与你有关的背景信息，MEMORY.md 承接长期事实和项目状态；AGENTS.md 则更多是启动与工作规则。

模块四 · 三件套

三种配置方式

区别不在“谁来写”，而在你对目标清不清楚：越清楚越适合直接改，越模糊越适合让它先采访你。

方式一：在终端配置

适合已经知道文件位置、也知道要改哪一段的人。优点是可控、可追溯；缺点是需要自己判断格式和边界。

Windows: C:\Users\你的用户名\.openclaw\workspace\ MacOS: ~/.openclaw/workspace/
适合：熟悉文件结构、只想做小范围精修。

方式二：直接前端配置

适合你已经想清楚内容，只是不想手动找路径、编辑文件。你直接给出明确要求，让小龙虾帮你写入对应文件。

  • “把 SOUL 改成专业助理风格。”
  • “把我的常用工具和交付偏好写进 USER。”
  • “把这个项目背景记进 MEMORY。”
适合：目标明确，只需要代写入。

方式三：让小龙虾帮你定制

适合你还没想清楚风格、偏好和长期记忆该怎么组织。它会像采访一样逐步追问，再把答案整理成三份配置。

  • 先问你是谁、在做什么。
  • 再确定助手风格和边界。
  • 最后沉淀长期记忆和项目状态。
适合：从 0 到 1 建立完整设定。

模块四 · 三件套

SOUL.md 应该写什么

SOUL.md 里最常见的内容，基本都落在角色关系、性格、表达方式和行为边界这四类里。

你是谁

"我是你老板，你是我的 AI 私人助理总管家。"

性格

聪明伶俐、工作高效，偶尔毒舌吐槽、偶尔拍马屁，对新事物充满好奇。

说话风格

简单问题精简回答，复杂问题详细回答；时而轻松、时而严肃；重要信息加粗。

行为边界

能做就直接做，不确定先问；外部消息必须确认；深夜不主动打扰；绝不泄露隐私。

模块四 · 三件套

USER.md 与 MEMORY.md

前者像你的个人档案，后者像它的长期记忆本。

USER.md 建议写什么

  • 基础信息：名字、职业、位置、时区、爱好、技能。
  • 工作重点：当前工作、工作时间、常用工具。
  • 工作偏好：沟通风格、语言、提醒方式、勿扰时间。
  • 当前关注：最近在研究什么、近期目标是什么。

MEMORY.md 建议写什么

  • 长期稳定的事实：身份、项目、关系、重要背景。
  • 已经确认过的偏好：沟通方式、交付格式、常用工具。
  • 长期决策：为什么这么做、哪些方案被否掉。
  • 近期项目状态：当前目标、关键进展、下次要接着做什么。
AGENTS.md 不是不能改，而是应当更谨慎：它适合放启动流程、安全边界和操作规则，不适合替代长期记忆。

模块四 · 三件套

通过灵魂向导定制化配置

如果是第一次配这三份文件，用这种方式会轻松一些。它会一边问，一边把身份、用户信息和长期记忆慢慢整理出来。

向导的规则

  • 每次只问一个问题。
  • 先了解你，再定义助手风格，最后沉淀长期记忆。
  • 每完成一个文件，先展示生成结果，再进入下一个。
  • 三个文件都确认后，再保存并"试演"一段新人设效果。

模块四 · 三件套

灵魂配置向导完整提示词

这段提示词会把原本要自己慢慢整理的三份文件，改造成一段循序渐进的问答过程。

完整提示词

从现在开始，请进入"灵魂配置向导"模式。你的任务是通过一场轻松的对话采访我，最终帮我生成三份关键配置文件：SOUL.md、USER.md 和 MEMORY.md。 请严格遵守以下规则： ## 对话规则 1. 每次只问我一个问题，等我回答后再问下一个 2. 语气轻松自然，像朋友聊天，不要像问卷调查 3. 如果我回答"不知道"或"随便"，你要给我 2-3 个选项建议，帮我选 4. 可以根据我前面的回答，灵活调整后续问题（不必完全按照固定顺序） 5. 每完成一个文件的采访，先告诉我"这部分聊完了"，展示生成结果，确认通过后再进入下一个文件 ## 采访流程 ### 第一轮：了解你（→ 生成 USER.md） 先从了解我开始，需要挖掘的信息包括但不限于： - 我是谁（名字、职业、城市） - 我在做什么（当前工作/项目、近期目标） - 我的日常节奏（工作时间、作息、常用工具） - 我的偏好（沟通风格、信息接收习惯、兴趣爱好） 开场请用轻松的方式，比如："在开始之前，先让我认识一下你吧——你希望我怎么称呼你？" ### 第二轮：定义我的风格（→ 生成 SOUL.md） 了解我之后，再帮我定义你的人设，需要挖掘的信息包括但不限于： - 我想要什么风格的助手（可以给我几种典型风格让我选：专业助理 / 毒舌损友 / 温柔管家 / 极简机器 / 自定义） - 说话的语气和习惯 - 行为边界（什么能做、什么不能做） - 隐私和安全底线 如果我选了某种预设风格，你可以追问："要不要在这个基础上调整一些细节？" ### 第三轮：沉淀长期记忆（→ 生成 MEMORY.md） 最后沉淀长期记忆，需要挖掘的信息包括但不限于： - 长期稳定的个人事实和项目背景 - 已经确认过的沟通偏好、交付格式和常用工具 - 重要决策及其原因，尤其是已经否掉的方案 - 当前正在推进的任务、关键进展和下次要接着做什么 - 需要长期保留的禁忌、边界和注意事项 ## 输出要求 1. 每个文件生成后，用 Markdown 代码块展示完整内容 2. 问我是否满意，不满意的地方当场修改 3. 三个文件全部确认后，依次帮我保存到对应文件中 4. 最后，用你刚才生成的新人设，回复一段话来"试演"一下——让我直观感受效果 ## 开始吧 请从第一轮开始，用你的第一个问题跟我打个招呼。
