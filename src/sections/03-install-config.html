Provider 凭证准备

这一页只讲一件事：在进入 onboarding 之前，先准备好可用的模型密钥。常见做法分成 API Key 和 Token Plan 两种。

方案 A：先用 API Key

  • 适合首次安装、功能测试、低频使用。
  • 优点是开通快、试错成本低，遇到问题也更容易排查。
  • 拿到 API Key 后，后面的 onboarding 填写方式最直接。

方案 B：长期用 Token Plan

更适合长期运行、固定工作流和高频调用。通常要先开通套餐，再到平台里拿可用的接口密钥。

如果你已经确认会长期使用 OpenClaw，再考虑 Token Plan 会更合适。

第四部分 · 步骤 01 补充

API Key 和 Token Plan 有什么区别？

它们最后都能给 OpenClaw 提供模型凭证，核心区别不在能不能接入，而在付费方式、使用频率和长期成本控制。

对比项 API Key Token Plan
本质 普通接口密钥 订阅或套餐模式下使用的接口凭证
适合谁 刚开始体验、低频使用、新手测试 长期使用、重度使用、团队稳定接入
成本特点 按量消耗，先用先试 更适合长期规划，成本预期更稳定
灵活性 高，开通快，试错成本低 更适合固定工作流和长期运行
常见场景 跑通安装、测试功能、偶尔使用 日常高频对话、定时任务、长期托管

第一次部署

建议先用 API Key，把安装、onboarding 和 Dashboard 验收完整走通。这样上手最快，遇到问题也最容易定位。

长期运行

当本地 Gateway、渠道接入和自动化任务都稳定后，再评估 Token Plan 是否更适合长期成本和调用频率。

第四部分 · 步骤 02

执行安装命令

OpenClaw 官方推荐先使用安装脚本。脚本会完成 CLI 安装，并进入 onboarding 流程。

macOS / Linux

curl -fsSL https://openclaw.ai/install.sh | bash

在终端中直接执行上面的命令。

Windows（PowerShell）

Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned iwr -useb https://openclaw.ai/install.ps1 | iex

先放开当前用户脚本权限，再执行安装命令。Windows 原生可用，但更推荐 WSL2。

安装完成后，应继续执行 `openclaw onboard --install-daemon`；如果脚本已自动进入向导，可直接继续下一步。
如果一键安装命令失败，请直接跳到附录 A：手动安装，按操作系统分支逐步排查环境与依赖。

第四部分 · 步骤 03 速查

onboarding 推荐选项

首次本地部署时，下面这组选择通常最容易跑通主流程。

命令

`openclaw onboard --install-daemon`

选择

保留 `--install-daemon`，让 Gateway 作为后台服务安装。

向导模式

进入 onboarding 后，优先选择 `Quick start`。

作用

优先走官方默认配置，减少首次安装时的自定义变量。

完成设置

渠道先跳过，Skills 先选 `No`，交互入口优先选 `Web UI`。

顺序

先完成本地可用，再进入飞书接入和 Skills 安装。

第四部分 · 步骤 05

部署验收

部署完成后，按命令行检查 → 服务状态 → 控制界面 这条顺序验收。

验收命令

openclaw doctor openclaw gateway status openclaw dashboard

`openclaw --version` 可先确认 CLI 是否安装成功；`doctor` 用于检查依赖与配置；`gateway status` 用于查看 Gateway 是否已运行；`dashboard` 用于打开控制界面。

验收完成时，应同时满足三件事：CLI 可调用、Gateway 正在运行、Dashboard 能正常打开。

第四部分 · 步骤 06

故障定位顺序

出现异常时，按固定顺序排查，可快速判断问题位于环境、服务、模型还是渠道层。

定位顺序

1

版本是否满足：`node -v`、`git --version`、`openclaw --version`。

2

环境是否健康：`openclaw doctor` 看依赖、路径和基础配置。

3

服务是否在线：`openclaw gateway status` 看 Gateway 有没有真正跑起来。

4

Provider 是否可用：检查 API Key、余额、模型名和地区访问限制。

常见现象与定位

  • 命令都不存在：先回到安装环节，不要继续往后配。
  • Gateway 没启动：先解决服务问题，不要急着看飞书和 Skills。
  • 页面打开但不能回复：优先查 Provider 凭证和模型配置。
  • 本地可用、飞书不可用：说明核心安装已过关，问题已经缩小到渠道接入层。
排查完成后，应能明确判断问题属于哪一层，再进入对应部分继续处理。
