# OpenClaw 前三部分讲稿

## 第一部分：认识 OpenClaw

### 一、从聊天窗口到真实工作流

大家过去接触 AI，大概率都是从一个聊天窗口开始的。

ChatGPT、Claude、DeepSeek、Kimi、通义千问这类工具，大家都不陌生。打开网页或 App，输入一句话，AI 就能回答问题、写文案、总结文章、翻译内容、修改代码、生成方案。对普通用户来说，这已经带来了很明显的效率提升。

比如，写通知，可以让 AI 帮忙润色；写日报周报，可以让 AI 帮忙整理结构；看一篇长文章，可以让 AI 提炼重点；遇到一个陌生概念，可以让 AI 用更简单的方式解释；写代码遇到报错，也可以把错误信息贴进去，让 AI 帮忙分析。

这就是我们最熟悉的 <u>对话型 AI</u>。用户把问题交给 AI，AI 在聊天窗口里生成回答。问答、写作、总结、辅助思考，这些事情它已经做得很顺手。

放到真实工作里，问题就出来了。

**第一个问题：上下文搬运。**  
真实任务往往带着大量背景信息。比如一份会议纪要，可能要结合会议聊天记录、参会人员、项目背景、历史决策和后续安排。用户要让 AI 完成这个任务，通常得先把这些材料整理好，再复制到聊天窗口里。材料越多，任务越复杂，前面的准备成本就越高。

**第二个问题：结果转移。**  
AI 生成内容以后，用户还要把结果复制回真实工作环境。会议纪要要放回飞书文档，待办要放进任务系统，日报要发到群里，代码要回到项目目录。AI 生成了内容，但后面的落地动作还得人来做。

**第三个问题：任务持续性弱。**  
很多 AI 对话都是一次性的。今天让它总结文章，明天让它写周报，后天让它整理会议记录，每一次都像重新开始。它很难持续关注一个入口里的新消息，也很难按时间自动处理任务，更难围绕一个长期目标持续跟进。

**第四个问题：工作现场分散。**  
真实工作很少只发生在一个聊天窗口里。它可能同时发生在飞书消息、文档、表格、浏览器、本地文件、内部系统、代码仓库、命令行脚本和自动化平台中。AI 如果只待在聊天窗口里，就很难碰到这些现场。

**第五个问题：工具使用受限。**  
很多时候，我们希望 AI 能读取文件、连接外部系统或服务、运行预先写好的自动化操作、发送消息、更新文档、整理表格、触发固定流程。对话型 AI 能给出建议，真正改文件、发消息、更新系统，通常还得人来做。

真实工作流对 AI 提出了新的要求：

- AI 要出现在用户原本使用的入口里，比如飞书、Telegram、Slack、微信、网页控制台或者团队协作工具。
- AI 要理解文件、消息、历史资料、项目背景和用户偏好。
- AI 要能连接 API、脚本、数据库、网页服务和业务系统。
- AI 还要把结果交付到用户真正使用的位置。

OpenClaw 要解决的，就是这类从聊天窗口进入真实工作流的问题。

### 二、Agent：从回答问题到执行任务

要理解 OpenClaw，先看一个词：<u>Agent</u>。

Agent 是围绕目标推进任务的 AI 系统。它的重点在于使用工具、推进过程、交付结果。

普通对话型 AI 更偏向生成回答。用户问一个问题，它回答一个问题；用户给一段材料，它生成一段内容。Agent 会往前多走一步：根据用户目标组织步骤、选择工具、执行动作，再把结果交付到指定位置。

可以先这样理解 Agent：会使用工具、能推进任务的 AI 助手。

它通常要做几件事。

先理解目标。用户说“帮我整理一下今天会议的重点”，Agent 要判断用户真正想要的是会议结论、行动项、负责人、截止时间，还是一份可以直接发给团队的会议纪要。

再拆解任务。复杂任务通常不能一步完成。比如生成会议纪要，可能要先获取会议材料，再识别主要讨论，再整理决策点，再提取待办事项，最后输出成文档格式。

然后选择工具。可能是读文件、查网页、调用飞书接口、操作表格、运行脚本，也可能是读取数据库。

执行过程中，它还要根据中间结果继续行动。比如它读取了一份会议记录，发现里面涉及多个项目，就继续按项目拆分待办；发现某个负责人缺失，就把它标记为待确认。

最后，结果要回到用户真正使用的位置。可能是飞书消息、飞书文档、表格、邮件、项目系统，也可能是本地文件。

我们用会议纪要举例。

在传统对话型 AI 中，用户通常先复制会议内容，让 AI 生成纪要，再把结果复制回飞书或文档。这个流程里，AI 主要负责生成文本。

在 Agent 模式中，流程更接近真实工作：Agent 从飞书或指定入口接收任务，获取会议内容，整理会议结论、待办事项、负责人和截止时间，再将结果返回飞书或写入文档。任务从一次文本生成，变成了**获取材料、整理结构、提炼待办、交付结果**的一组动作。

再看出差计划。

对话型 AI 可以根据用户输入，生成一份出差计划文本。Agent 可以进一步读取日程、查询交通和酒店信息、整理出差待办、生成计划，并发送到指定入口。

这类任务都有几个共同点：步骤多，信息来源多，要用工具，结果还要交到某个具体位置。

真实工作流里的任务，经常就是这样一串动作：找资料，判断资料价值，整理结构，生成内容，调整格式，发送给指定对象，归档结果，后续继续跟进。

Agent 的价值，就在于把这些动作组织起来，让 AI 从回答问题走向执行任务。

### 三、OpenClaw 是什么

理解了 Agent，再看 OpenClaw。

OpenClaw 的出现，来自一个很朴素的需求：**AI 要进入真实工作现场。**

真实工作分散在很多地方。我们在飞书里沟通，在文档里写材料，在表格里整理数据，在浏览器里查资料，在本地目录里保存文件，在脚本和自动化工具里处理重复任务。一个真正有用的 AI 助手，要靠近这些地方。

OpenClaw 的创始人 Peter Steinberger 介绍过，这个项目最早来自一个周末项目，后来经历了命名变化，并在社区中快速扩散。官方博客里也记录了从 Clawd、Moltbot 到 OpenClaw 的命名过程。

从定位上看，OpenClaw 是一个 <u>自托管的个人 AI 助手系统</u>。简单说，就是它可以运行在用户自己的电脑或服务器上。它通过 Gateway 把聊天入口、模型能力、Agent、Skills 和工具连接起来。这里的 Gateway，可以先理解成 OpenClaw 的**中转和调度服务**：消息从飞书、Telegram 这类入口进来，Gateway 负责接住，再转给后面的 AI 助手处理。

这句话有点技术化。换成课堂里的说法：

大模型负责理解、推理和生成。

OpenClaw 负责接收消息、安排任务、连接各种能力。

飞书、Telegram、Slack、Web UI 这些入口，是用户和助手对话的地方。

Skills 像工具箱里的说明书，告诉 Agent 某种能力该怎么用。

本地文件、脚本、API 和自动化任务，构成 AI 真正干活的环境。

OpenClaw 做的事情，就是把这些部分组织成一个个人 AI 助手系统。

后面的课程会围绕四件事展开：**部署、飞书接入、Skills 配置、案例实践。**

部署解决系统怎么跑起来。飞书接入解决 AI 怎么来到工作入口。Skills 配置解决能力怎么扩展。案例实践则用真实任务看看它能帮我们做到什么程度。

### 四、OpenClaw 为什么适合真实工作流

OpenClaw 适合真实工作流，可以从三个角度理解：入口、能力、持续运行。

#### 1. 入口价值：在用户原本的地方使用 AI

很多工作任务本来就从消息入口产生。

团队在飞书群里讨论项目进度，老师在群里通知课程安排，运营人员在群里同步活动方案，项目负责人在群里安排任务，学生在群里提问。消息入口本身就是一个高频工作入口。

官方文档中提到，OpenClaw 的 Gateway 可以接入多种消息入口。除了 WebChat 和移动端入口，也支持 Discord、Google Chat、iMessage、Matrix、Microsoft Teams、Signal、Slack、Telegram、WhatsApp、Zalo 等工具。

这门课选择飞书作为主要实操入口，因为飞书在国内团队协作场景中很常见，而且同时具备消息、文档、表格、日历、任务等能力。OpenClaw 的 Feishu 文档中也说明，飞书入口通过机器人接收消息变化，当前版本已经内置 Feishu 插件，无需单独安装。

对用户来说，AI 可以直接待在原来的工作入口里。比如在飞书里说一句：“帮我把这段讨论整理成会议纪要。”系统就可以围绕这条消息继续处理。

#### 2. 能力价值：把模型能力连接到文件、工具和系统

只接入消息入口还不够。AI 要完成任务，还需要能使用资源。

这些资源包括本地文件、本地目录、脚本、API、数据库、网页服务、自动化平台和业务系统。OpenClaw 的 Gateway 负责把入口、Agent、模型和工具组织起来。官方架构文档中提到，一个长期运行的 Gateway 会管理不同的消息入口，命令行、Web UI、自动化任务和其他节点都可以连接到它。

在这里，OpenClaw 更像一个<u>中间调度层</u>。用户从飞书发出任务，OpenClaw 接住任务，再交给 Agent。Agent 结合模型判断要用什么能力，然后调用文件、工具、Skills 或外部服务。

Skills 是这里很重要的扩展机制。官方 Skills 文档说明，OpenClaw 通过一个个 Skill 文件夹教会 Agent 使用工具。每个 Skill 都是一个目录，核心文件是 `SKILL.md`，里面写着这个能力什么时候用、怎么用。

口语一点讲，**Skill 像一份能力说明书。** 它告诉 Agent：什么时候用这个能力，怎么用，需要什么输入，能输出什么结果。

有了 Skills，OpenClaw 就能进入更具体的工作场景。比如文件整理、飞书通知、网页监控、课程资料、日报周报、信息检索。

#### 3. 持续价值：把一次性问答沉淀为长期助手

真实工作经常具有持续性。

比如每天早上收集行业新闻，下午生成课程资料，晚上总结当天工作；每周生成项目周报；每次群里开完会后自动整理会议纪要；每次某个信息源更新后自动推送提醒。

OpenClaw 可以长期运行在个人电脑或服务器上，通过 Gateway 管理会话、入口和工具。官方文档中也提到，Gateway 负责统一管理会话、消息分发和入口连接，控制界面、命令行和 Web UI 都可以连接到它。

长期助手还涉及记忆和上下文。OpenClaw 的 memory 文档中提到，Agent 有 `MEMORY.md`、每日 notes 和可选的 `DREAMS.md` 等记忆相关文件。`MEMORY.md` 用来记录比较稳定的偏好、事实和决策。

OpenClaw 这类系统适合做长期助手。它可以逐渐熟悉用户的工作方式，处理那些反复出现的信息任务。

### 五、OpenClaw 能用来干什么

我们用几个具体场景看 OpenClaw 的用途。

#### 1. 消息与协作助手

第一类场景，是消息与协作助手。把 OpenClaw 放进飞书这样的消息入口，让它围绕群聊、会议和项目同步来帮忙。

用户可以直接在飞书里提问：“帮我把这段讨论整理成会议纪要。”也可以说：“根据刚才的聊天内容，整理出三个行动项。”还可以说：“帮我写一段项目进展同步，语气正式一点，适合发到团队群。”

这里的价值很直观：材料在消息入口里，AI 也在消息入口里，结果仍然回到消息入口里。

飞书插件文档里列出了很多能力，包括消息读取与发送、文档创建与更新、表格编辑、日历事件管理和任务管理等。飞书接入的价值，体现在 AI 可以接触团队协作里的消息、文档、表格、日历和任务。

这一类场景可以覆盖即时问答、群聊总结、会议纪要、项目同步、通知生成和行动项整理。这些能力通常依赖飞书接入、权限配置和相应工具，安装完成只是第一步。

#### 2. 资料整理与内容生成

第二类场景是资料整理。

比如我们准备一门课，手头可能有课程大纲、教学案例、产品文档、参考文章、实操流程、常见问题。在提供资料或配置资料访问权限后，OpenClaw 可以帮助把这些资料整理成课程结构、讲稿、讲义、案例脚本和课堂练习。

再比如团队准备一次项目汇报，可能要汇总群聊讨论、历史文档、项目里程碑、数据表格和负责人反馈。在资料准备充分的情况下，OpenClaw 可以辅助整理出汇报大纲、项目进展、风险点和下一步计划。

这类场景的重点，是把零散材料整理成结构化内容。输出可以是大纲、表格、清单、报告、会议纪要、课程笔记，也可以是行动项。

对普通学员来说，这类场景最容易上手，因为它和大家熟悉的 AI 使用方式接近。OpenClaw 更进一步，把资料来源、输出位置和重复流程串起来。

#### 3. 本地文件与个人知识助手

第三类场景，是本地文件与个人知识助手。

很多人的资料都放在本地目录里：项目文件、课程资料、学习笔记、论文材料、合同模板、脚本文件、图片素材、历史报告。普通 AI 工具通常需要用户主动上传或复制内容。OpenClaw 这类自托管助手系统，更适合围绕本地环境组织工作。

在配置本地权限和相应工具后，用户可以让它帮忙定位某个项目文件，总结某个目录下的文档，整理一批材料的主题，或者根据项目资料生成说明文档。

OpenClaw 的 Agent 运行文档中提到，它会使用一个固定的工作目录来放工具和上下文；新会话启动时，还会加载一组用户可以编辑的启动配置文件，例如 `AGENTS.md`、`SOUL.md`、`TOOLS.md`、`IDENTITY.md`、`USER.md` 等。这些文件可以告诉助手如何工作、用户偏好是什么、能用哪些工具。

这意味着，OpenClaw 有自己的工作空间，可以围绕用户配置、工具说明、角色设定和本地上下文运行。它是在一个配置好的工作空间里处理任务。

本地文件能力再往前走一步，就会变成个人知识助手。比如一名教师可以用它积累课程材料，一名研究人员可以用它整理论文笔记，一名运营人员可以用它维护活动资料，一个小团队可以用它沉淀项目经验。这里同样要依赖资料、权限、记忆和规则配置。

#### 4. 自动化与长期助手

第四类场景，是自动化与长期助手。

比如每天早上自动收集行业新闻，整理成摘要后发送到飞书；每周五生成项目周报；定时检查某个网页是否更新；每天把学习记录整理成清单；把一批材料定期归档到指定目录。

这类任务的特点是重复、固定、有明确输出位置。它们很适合沉淀成自动化流程。

OpenClaw 的配置文档中提到，常见配置包括连接消息入口、控制谁可以给机器人发消息、设置模型和工具、配置权限隔离和自动化任务，以及调整会话、媒体、网络或界面相关选项。

OpenClaw 可以围绕入口、模型、工具和自动化配置长期运行。普通用户可以先从简单任务开始，比如每天生成资讯摘要，或者每次群聊讨论后生成待办清单。

长期助手的价值，在于积累用户的偏好、背景和历史资料。OpenClaw 的 memory 文档中，`MEMORY.md` 用于保存长期记忆，日记文件用于记录当天和前一天的上下文。

一次性 AI 工具像临时帮手，长期助手更像熟悉你工作方式的助理。

如果把它放到团队协作里，还要特别注意权限。飞书插件文档明确提醒，授权后 OpenClaw 会在授权范围内以用户身份行动，可能带来敏感数据泄露或未授权操作等风险；文档也建议把连接 OpenClaw 的飞书机器人作为私人会话助手使用，谨慎加入群聊。

后面实操时，我们先用私聊场景跑通，再扩展到群聊；先开必要权限，再增加高级能力。

### 六、OpenClaw 为什么被社区关注

讲完 OpenClaw 能做什么，再看它为什么会被社区关注。

这个故事可以从 2025 年 11 月讲起。

最早的时候，它还带着很强的个人实验色彩。公开资料里提到，Peter Steinberger 一开始做的是一个类似 WhatsApp Relay 的东西：人通过消息软件发一句话，背后有一个 AI 助手去处理事情。这个方向很直观。大家每天已经在微信、飞书、Telegram、WhatsApp 里工作和沟通。如果 AI 可以待在这些入口里，它就离真实工作近了一步。

2025 年 11 月 24 日，OpenClaw 的 GitHub 仓库创建。早期项目名叫 Clawdbot，名字里带着 Claude 和 claw 的影子，也带着后来那套“龙虾”梗。这个阶段的重点，是把 AI 从网页对话框里拉出来，让它通过消息入口、文件、工具和本机环境做事。

到 2026 年 1 月，故事开始变得热闹。1 月 27 日，项目改名为 Moltbot。Molt 是龙虾换壳的意思，这个名字很有梗。三天后，1 月 30 日，项目又改名为 OpenClaw。这个名字稳定下来以后，社区传播明显加速。

这里有一个很有意思的爆点：OpenClaw 的爆火，和 Moltbook 这类实验一起出现。有人用 OpenClaw 代理去搭建面向 AI Agent 的社交网络，很多人第一次意识到，Agent 已经可以自己发帖、协作、生成内容，甚至参与一个“机器之间的社交场”。这件事听起来有点怪，也正因为怪，它很容易传播。

到 2026 年 2 月，OpenClaw 已经进入大众技术媒体视野。Tom’s Guide 在 2 月 4 日报道时提到，它已经超过 16 万 GitHub stars。一个开源 AI 助手，几个月时间冲到这个量级，说明它击中了很多人的想象：我想拥有一个真正待在身边、能接消息、能看文件、能跑工具、能长期工作的 AI 助手。

再看 OpenClaw 当时的热度。

公开资料显示，2026 年 3 月 2 日，OpenClaw 已经达到 24.7 万 stars、4.77 万 forks。到了 2026 年 4 月 24 日晚上，我刚查 GitHub API，OpenClaw 已经有 363,267 stars、74,279 forks，open issues 达到 19,172 个。

## 第二部分：OpenClaw 的基本运行原理

### 一、先从一条消息看起

上一部分我们讲到，OpenClaw 要解决的是 AI 进入真实工作现场的问题。

现在往系统内部看。

OpenClaw 的运行可以从一条消息开始理解。用户在飞书、Web UI、Telegram 或 Slack 里发出一句话，这句话进入 OpenClaw。系统先判断消息来自哪里，属于哪个会话，当前能用哪些能力，然后把任务交给 Agent。Agent 结合模型、工具、Skills、当前上下文和长期记忆处理任务。处理完成后，结果回到用户原来的入口。

这条链路里有五组概念，后面会反复用到。

<u>Channel</u> 是外部入口的适配层。飞书、Web UI、Telegram、Slack 都有自己的消息格式和事件机制。Channel 把这些外部消息转成 OpenClaw 内部能处理的事件。

<u>Gateway</u> 是中转和调度服务。它长期运行在本机或服务器上，接收不同 Channel 的消息，维护连接，识别会话，检查权限，把任务交给合适的 Agent，再把结果送回原入口。

<u>Agent</u> 是任务执行者。它围绕用户目标准备上下文，请求模型判断，发起工具调用，读取工具结果，再整理输出。

<u>Model、Tools、Skills</u> 是能力层。模型负责理解和生成。Tools 负责具体动作，比如读文件、访问网页、执行脚本、调用 API、发送消息。Skills 是任务说明书，写清某类任务的步骤、材料要求和输出格式。

<u>Workspace、Context、Memory</u> 是上下文层。Workspace 是资料和文件所在的位置。Context 是本轮任务里模型能看到的信息。Memory 是跨任务保留下来的长期事实、偏好和项目背景。

后面的部署、飞书接入、Skills 配置和案例实战，都可以放回这条链路里看。

---

### 二、OpenClaw 在系统里的位置

看这张分层图。

最上面是用户。用户面对的是熟悉的入口，可能是飞书机器人，可能是 Web UI，也可能是 Telegram、Slack 这类消息工具。

入口下面是 Channel。不同平台发来的消息，字段、鉴权方式、事件类型都不一样。飞书有飞书的事件，Telegram 有 Telegram 的更新，Web UI 有网页端输入。OpenClaw 先把这些来源统一成内部事件，后面的 Agent 才能按同一套方式处理任务。

Channel 下面是 Gateway。Gateway 是 OpenClaw 的控制面，负责组织任务怎么走。谁发来的消息，属于哪个会话，用哪个 Agent，能否调用某个工具，结果发回哪里，这些都要经过 Gateway。

Gateway 下面是 Agent。用户说“帮我整理会议纪要”，Agent 要把这句话变成一组动作：读取材料，识别会议主题，提取结论，整理待办，形成输出。任务需要外部能力时，Agent 会调用文件系统、网页、脚本、飞书或其他服务。

Agent 下面是模型、工具和 Skills。模型做判断，工具做动作，Skills 给方法。比如会议纪要要提取主题、结论、行动项、负责人和时间；日报要按固定格式输出；课程讲稿要适合口播。这些方法都可以沉淀到 Skills 里。

最下面是 Workspace、Context 和 Memory。它们决定 Agent 手里有什么材料。Workspace 提供文件和项目资料，Context 提供本轮可见信息，Memory 提供长期偏好和历史事实。

OpenClaw 是一个 Agent Gateway。它把消息入口、模型能力、工具系统和长期资料连接成一条任务链路。

---

### 三、几个角色分开看

把角色再拆细一点。

Channel 处理消息入口。飞书消息怎么进来，群聊里谁 @ 了机器人，Web UI 输入来自哪个会话，这些都属于 Channel 层。后面做飞书接入，就是让飞书 Channel 能把消息稳定送进 OpenClaw。

Gateway 处理运行时调度。它关心连接、会话、权限、任务分发和结果回传。Gateway 状态异常时，外部入口可能已经配置好了，消息仍然卡在接入或分发环节。部署验收时，我们会反复看 Gateway 是否启动，是否能被控制界面看到。

Agent 处理任务过程。一次用户请求可能只有一步，也可能包含多次模型判断和多次工具调用。Agent 要在模型的推理结果和真实工具结果之间来回切换。工具执行失败、材料不足、权限不够、上下文太长，都会影响后续判断。

Model 处理理解、推理和表达。模型通过当前 Context 判断下一步动作，或者生成最终回复。系统能力同时来自入口、工具、Skills 和上下文。

Tools 和 Skills 处理能力扩展。Tools 是可执行动作，Skills 是做事方法。读一个文件是 Tool。把一份会议记录整理成结构化纪要，是 Skill 里可以写清的方法。好的 Skill 会把任务步骤、材料要求、检查点和输出格式写清楚，让 Agent 在类似任务里少靠临场发挥。

Workspace、Context、Memory 处理上下文。Workspace 是资料所在的位置，Context 是本轮模型能看到的内容，Memory 是之后还能继续使用的信息。排查问题时分三类看：资料位置、Context 装载、Memory 检索。

---

### 四、一条消息如何被处理

看一条消息的完整运行链路。

用户先在入口里发出任务。比如在飞书里发：“帮我整理一下刚才这段会议讨论。”这条消息先进入飞书 Channel。

Channel 把飞书事件转成 OpenClaw 内部消息事件。这个事件会带上来源信息，比如用户 ID、群组 ID、会话 ID、入口类型、消息文本，有时还会带附件或引用消息。Gateway 后面要靠这些信息判断上下文归属。

消息进入 Gateway 后，先做来源识别。它要知道消息来自哪个入口、哪个用户、哪个群组、哪个会话。接着做策略检查。这个用户是否有权限，群聊里是否需要 @ 机器人，当前入口是否允许触发 Agent，当前 Agent 能使用哪些模型和工具，都在这个阶段判断。

策略通过以后，Gateway 把任务交给 Agent。

Agent 开始准备 Context。它会把当前用户输入、会话历史、系统规则、可用工具描述、相关 Skill 说明、Workspace 里的材料、Memory 检索结果组合起来，形成模型本轮能看到的输入。

模型拿到 Context 后开始判断。简单任务可以直接回复，比如润色一句话。复杂任务会生成工具调用请求，比如读取文件、搜索资料、调用飞书能力、执行脚本或访问网页。

Agent 收到工具调用请求后执行工具。工具返回结果后，结果进入 Context，模型继续判断。这个过程可能重复好几轮。

Agent 形成输出后，Gateway 把输出转换成原入口能接收的格式，再交给 Channel 发送回去。用户看到的是一条回复，系统内部已经走完事件接入、Session 路由、Context 组装、工具执行和响应回传。

Session 这个词要记住。

Session 是一次持续对话或任务上下文的容器。它让系统知道，当前消息和之前哪些消息属于同一段对话。用户追问“那把第二条改短一点”时，系统才能知道“第二条”指的是什么。

---

### 五、Gateway：中转和调度服务

Gateway 是第二部分里必须讲清楚的概念。

OpenClaw 在入口和模型之间放了一个长期运行的 Gateway。它像系统的调度台，所有入口消息先到这里，所有任务分发也从这里出去。

Gateway 先做入口适配。飞书、Web UI、Telegram、Slack 发来的消息格式都不一样。Gateway 通过 Channel 层拿到统一事件后，后面的 Agent 才能按相同方式处理任务。系统新增入口时，主要改入口适配层，Agent 的任务处理方式可以保持稳定。

然后是 Session 路由。Gateway 要判断一条消息进入哪个 Session。私聊、群聊、网页会话、不同用户、不同 Agent，都可能对应不同上下文。路由错了，轻则上下文混乱，重则把一个用户的内容带到另一个会话里。

再往下是策略检查。OpenClaw 具备实际行动能力，能读文件、跑命令、访问网页、发送消息。越接近真实工作，越需要控制边界。Gateway 会参与权限、白名单、可用模型、可用能力集合等判断。后面接飞书时，群聊、私聊、机器人权限、回调地址都要认真配置。

最后是响应回传。Agent 输出还要适配每个平台。飞书有飞书的消息格式，Web UI 有 Web UI 的显示方式，文档更新也有自己的接口要求。Gateway 要把 Agent 输出包装成目标入口能接受的形式。

部署 OpenClaw 时，Gateway 正常运行是基本前提。飞书接入时，飞书消息要能送到 Gateway。Skills 配置后，Gateway 和 Agent 要能识别可用能力。排查问题时，也经常从 Gateway 状态开始看。

---

### 六、Agent 的执行循环

Agent 的运行可以拆成四个动作：Context assembly、Planning、Tool execution、Result synthesis。

Context assembly 是上下文组装。Agent 先准备相关信息，包括用户当前输入、会话历史、系统提示、可用工具描述、相关 Skills、Workspace 里的材料、Memory 检索到的长期信息。准备得好，模型判断就更稳；准备得差，模型只能根据不完整材料猜。

Planning 是任务判断和步骤规划。模型会根据 Context 判断当前任务类型。它可能直接生成回复，也可能决定调用工具。用户说“整理这份文档里的行动项”，模型要判断需要读取文档。用户说“把结果发到群里”，模型要判断需要调用消息发送能力。用户说“按我们上次的格式来”，模型要依赖会话历史或 Memory。

Tool execution 是工具执行。模型提出工具调用请求后，Agent 运行具体工具。工具可以是文件读取、命令执行、网页访问、API 调用、消息发送、文档写入。工具返回结果以后，Agent 会把结果放回任务过程里，让模型继续处理。

Result synthesis 是结果合成。模型吸收工具结果，把中间信息整理成用户需要的输出。这个输出可能是一段回复、一份清单、一张表格、一份报告、一段代码，也可能是下一步动作的说明。

任务复杂时，这四个动作会循环。先组装上下文，再做判断，再调用工具，再吸收结果。工具结果带来新信息，模型再做下一步判断。

以整理会议纪要为例。Agent 先拿到飞书消息和会话上下文。模型判断这是会议纪要任务，于是使用对应 Skill。Skill 里写着要提取会议主题、结论、行动项、负责人和截止时间。Agent 需要读取更多聊天记录时，就调用消息或文档工具。工具返回材料后，模型再整理结构，生成适合飞书发送的纪要。

普通聊天助手主要回答这一轮输入。Agent 会围绕目标推进过程，并根据工具返回结果继续行动。

---

### 七、Tools、Skills 与 Plugins

OpenClaw 的能力扩展分三层：Plugin、Skill、Tool。

Plugin 是安装包层。它可以带来一组能力，比如新的入口、新的工具、新的模型接入方式，或者一组 Skills。

Skill 是任务规则层。一个 Skill 通常是一个目录，核心文件是 `SKILL.md`。这个文件会写清楚技能名称、适用任务、触发条件、执行步骤、输入材料、输出格式，有些还会带参考文件、脚本或模板。

Tool 是执行层。它做具体动作。读文件、写文件、执行命令、访问网页、调用 API、发送消息，这些都接近 Tool 的范围。

三者放在一起看：Plugin 把能力装进系统，Skill 告诉 Agent 什么时候用、怎么用，Tool 把动作真正做出来。

用会议纪要举例。

飞书相关 Plugin 可能提供飞书入口、消息读取、文档写入等能力。会议纪要 Skill 会告诉 Agent，整理会议时要提取主题、结论、待办、负责人和时间。文件读取、消息获取、文档写入这些动作由 Tool 执行。

Skills 适合沉淀经验。

如果每次都靠用户在提示词里写“请你按主题、结论、行动项、负责人、截止时间来整理”，这套规则就停留在一次对话里。写成 Skill 后，类似任务就能复用同一套方法。课程场景也一样，讲稿怎么写、Slide 怎么写、学员手册怎么写，都可以沉淀成不同 Skills 或项目规则。

还要看边界。

Tool 和 Skill 越强，越要关注权限。读文件、发消息、执行脚本、调用外部 API 这些能力都可能带来风险。一个 Skill 写得粗糙，可能让 Agent 在错误场景里调用工具；一个第三方 Skill 来源不清，可能把危险指令带进系统。后面做 Skills 配置时，要同时看功能、来源、权限、适用范围和人工确认机制。

---

### 八、Workspace、Context、Memory

现在看 Workspace、Context、Memory。

这三个词决定 Agent 处理任务时手里有什么材料。

Workspace 是任务资料所在的位置，也就是 OpenClaw 的工作目录。配置文件、项目文档、课程资料、生成结果、一些长期可读写的文件，都可以放在 Workspace 里。Agent 处理任务时，会围绕这个工作空间读取、生成和修改资料。

Context 是模型本轮能看到的信息集合。

它包含用户当前输入、会话历史、系统规则、工具描述、Skill 内容、相关文件片段、工具返回结果、当前时间和运行环境。模型做判断时，依据的是进入 Context 的内容。资料要被检索、读取、筛选、组装进 Context，模型才能使用。

Context 还有一个限制：Context window。

每个模型都有上下文窗口，也就是一次能处理的 token 数量上限。材料太多时，系统就要做裁剪、压缩、摘要或检索排序。材料越贴近任务，模型越容易做对；内容越杂，模型越容易分心。

Memory 是跨任务保存的信息。

OpenClaw 的 Memory 更接近文件化记忆。长期偏好、项目背景、稳定事实、常用规则，可以写入记忆文件或日常记录。后续任务开始时，系统可以检索相关记忆，再把有用片段放进 Context。

Memory 中的内容需要经过检索和选择。进入 Context 的片段，才会影响本轮结果。

用课程资料整理举例。

Workspace 里可以放课程大纲、讲稿、Slide 源码、图片和实战手册。用户发来任务：“帮我把第二部分讲稿写得更像现场授课。”这句话进入当前 Session。Agent 会拿当前请求、相关历史、课程写作规则、第二部分大纲、第二部分 slide 文案，再结合 Memory 里保存的偏好，比如“讲稿要适合口播”“Slide 面向学员”“避免 AI 腔”。这些材料组成 Context，模型再开始写。

Workspace 文件位置错误，Agent 读不到资料。Context 缺少相关片段，模型只能泛泛地写。Memory 缺少偏好沉淀，后续任务就很难保持同一套风格。

长期助手依赖三个动作：资料放进 Workspace，相关内容进入 Context，长期偏好写入 Memory 并能在需要时被检索出来。

理解这三层，后面做实战会清楚很多。一个任务做得好不好，要看资料是否齐、上下文是否准、工具是否有权限、Skill 是否把方法写清楚。

## 第三部分：OpenClaw 的部署思路

### 一、先看 OpenClaw 准备在哪里干活

OpenClaw 常见部署路线有四类：本地电脑、云服务器、第三方平台托管、Docker。

这四类听起来像技术选型，实际先看使用场景。你准备让 OpenClaw 处理本机资料，还是接入飞书这类消息入口？你准备临时体验一下，还是让它长期在线？你愿意自己维护运行环境，还是想把环境交给平台？

先看 OpenClaw 要碰哪些资料。

资料都在自己电脑里，比如课程大纲、讲稿、实战手册、Slide 源码、项目文件，本地部署最顺。文件和 OpenClaw 在同一台电脑上，路径清楚，读写方便，出问题也容易查。

资料来自飞书、企业微信、钉钉、QQ 这类入口，并且希望机器人持续在线，云服务器更稳。服务器可以一直开着，公网访问和回调配置也更方便。

只想先看效果，第三方平台最快。平台把运行环境、入口和模型配置准备好，用户按页面流程创建实例，配置账号或额度，就能开始试用。

需要隔离环境、复现实验、测试多版本，Docker 会省事。它把 OpenClaw 放进容器里运行，部署、升级、清理都有边界。

再看 OpenClaw 要在线多久。

课堂跟做、个人学习、临时体验，本地电脑开着就够。学员完成安装、onboard、模型配置和基础验证，就能理解主流程。

长期机器人、定时任务、自动推送、消息监听，需要一台稳定运行的机器。云服务器和托管平台在这里更合适。个人电脑休眠、关机、断网都会中断服务。

最后看 OpenClaw 会拿到哪些权限。

OpenClaw 跑起来以后，可能接触文件、消息、API Key、脚本、浏览器、自动化任务和外部系统。部署越接近真实业务，权限边界越要提前想清楚。

这门课主线走本地部署。每个学员都能在自己的电脑上安装、配置、验证，马上看到 OpenClaw 的运行链路。云服务器、第三方托管和 Docker 放在部署思路里讲，方便大家课后按自己的场景选路线。

### 二、本地部署

本地部署，就是把 OpenClaw 安装在自己的电脑上运行。

macOS、Windows、Linux 都可以部署。常见流程是：准备系统环境，运行安装脚本，执行 onboard，配置模型，打开 Web UI 或 TUI 做验证。

onboard 是第一次启动时的引导配置。它会带你完成基础设置，让 OpenClaw 知道用什么模型、怎么启动、入口在哪里、配置是否完整。

Web UI 是网页界面，适合课堂大部分学员。能看配置、状态和交互入口。TUI 是终端界面，适合常用命令行的人，也适合服务器和远程运维场景。

本地部署最直接的好处，是控制权在自己手里。

文件、配置、日志、运行状态都在本机。出了问题，排查路线也清楚：安装是否完成，Gateway 是否启动，模型配置是否生效，Web UI 或 TUI 是否能打开。

它也方便处理本地资料。

比如这门课的资料就在一个项目目录里，有大纲、讲稿、实战手册、Slide 源码、图片、参考教程。OpenClaw 运行在本机，就能围绕这些文件做事。整理课程目录、生成讲稿、检查文档、修改文件，路径都很直接。

课堂里，本地部署的价值更明显。

学员亲手经历一遍安装、启动、配置模型、打开入口、发出第一条请求、看到回复，这个过程比听概念有用。OpenClaw 的运行链路会变得具体：Gateway 在跑，模型已接上，入口能收任务，结果能返回。

本地部署也有现实限制。

电脑要保持在线。电脑休眠、关机、网络断开，OpenClaw 就停。系统权限也会影响使用，尤其是读写文件、执行命令、访问本地目录这些操作。

课堂练习放在专门目录里，比如课程目录、测试目录、项目样例目录。文件访问范围先小一点。这样方便观察，也方便恢复。

本地部署适合几类人：

- 课堂学员
- 个人学习用户
- 主要处理本地文件和项目资料的人
- 想理解 OpenClaw 运行链路的人

先把本地部署跑通，再看云服务器、托管平台和 Docker，会轻松很多。底层链路一样：Gateway 要启动，模型要配置，入口要能收任务，结果要能返回。

### 三、云服务器部署

云服务器部署，就是把 OpenClaw 放到云厂商的服务器上运行。

常见选择包括腾讯云 Lighthouse、火山引擎、阿里云轻量服务器，也可以使用自己已有的云主机。部署方式可能是平台镜像、一键脚本，也可能是手动安装。

云服务器的价值，是长期在线。

本地电脑适合学习和本地资料处理。云服务器适合长期运行的助手。比如飞书机器人、企业微信机器人、QQ 机器人、定时收集信息、每天自动整理推送，这类任务都需要持续运行。

消息入口也更好配置。

接飞书时，经常会遇到事件订阅、回调地址、应用权限和消息回复。服务器有公网地址，飞书这类平台更容易把消息送到 OpenClaw。后面做飞书接入时，大家会看到：一个消息机器人要正常工作，入口、权限、回调、Gateway 状态都要对上。

云服务器还方便多设备访问。

OpenClaw 跑在服务器上，你可以从电脑访问，也可以从手机消息入口访问。它和某一台个人电脑的开关机关系更弱，更像一个长期在线的个人服务。

服务器也会带来维护工作。

配置、地域、网络、端口、安全组、系统版本、日志、备份、升级，都要有人管。端口开放要谨慎，后台管理入口要有认证，API Key 要保管好，日志里也可能出现敏感信息。

课堂讲云服务器，重点看判断方式。

目标是课堂学习、本机文件处理、个人体验，本地部署够用。目标是长期接收消息、每天自动运行任务、成为团队入口里的机器人，云服务器更合适。

已有服务器，成本主要是时间和维护精力。准备购买服务器时，要把服务器费用、运维成本和安全责任算进去。

云服务器部署适合几类人：

- 想做长期个人助手的人
- 需要消息机器人长期在线的小团队
- 有定时收集、整理、推送任务的人
- 熟悉云服务器基础操作的人

云服务器出问题时，按这条线查：服务器是否在线，OpenClaw 是否启动，端口是否可访问，模型配置是否正确，消息平台回调是否到达，权限是否完整，日志里有什么报错。

后面做飞书接入时，消息接收失败、机器人回复失败、权限报错，都能放回这条链路里定位。

### 四、第三方平台托管

第三方平台托管，就是由平台提供 OpenClaw 的运行环境。

用户通常在网页里创建实例，选择模型或额度，按平台流程接入入口，然后直接开始使用。常见平台包括飞书妙搭、扣子 OpenClaw、Kimi OpenClaw、WorkBuddy、JVSClaw、火山 ArkClaw、智谱 AutoClaw 等。

这类方案上手快。

很多平台已经准备好服务器、运行环境、入口集成和基础配置。用户省掉 Node.js、脚本安装、端口、进程管理这些前置工作，按步骤创建，就能看到一个能对话、能操作的 OpenClaw 或类 OpenClaw 入口。

它适合快速体验。

部署经验为零的用户，可以先绕开环境配置，直接体验消息入口、模型回复、技能市场、浏览器操作、文档处理等能力。

它也适合已经在平台里工作的用户。

一直在用飞书的人，可以先看飞书妙搭。已经在用 Kimi、扣子、WorkBuddy 的人，可以先从熟悉入口开始。账号、入口、模型、文档、权限体系都在同一个平台里，使用阻力会小一些。

托管平台也要看边界。

平台会决定可用模型、额度、价格、Skills 市场、工具权限、沙箱能力、数据保存策略、日志可见性和导出能力。普通体验时感受很轻，进入真实工作流以后，这些内容都要看。

处理个人学习资料，平台托管很方便。处理公司内部资料、客户信息、财务数据、业务后台，就要看平台的数据政策、权限设计和合规要求。

选第三方平台时，重点看五件事：

- 入口：是否接好飞书、浏览器、移动端、桌面端或 IM 工具
- 模型：是否支持你想用的模型，额度和价格是否能接受
- 工具：是否支持文件、网页、文档、表格、浏览器、自动化等能力
- 数据：数据保存在哪里，日志怎么看，资料能否导出
- 边界：权限怎么控制，沙箱怎么隔离，敏感操作是否需要确认

第三方平台托管适合几类人：

- 想快速体验 OpenClaw 的人
- 已经在使用飞书、Kimi、扣子、WorkBuddy 等平台的人
- 想减少服务器维护投入的人
- 需要课堂展示或快速演示的人

课堂里可以这样记：第三方托管最快看到效果，本地部署最方便理解细节，云服务器适合长期在线。

选方案时，创建速度只是一个指标。还要看它能接入哪些真实工作，权限能管到什么程度，费用能否接受，资料和结果能否拿出来。

### 五、Docker 部署

Docker 部署，就是把 OpenClaw 跑在容器里。

容器就是一个相对独立的运行环境。OpenClaw 需要的依赖、服务、端口、环境变量、数据目录，都可以放进一套容器配置里管理。常见方式是 Docker Compose，把多个配置写在一起，一条命令启动服务。

Docker 的优势是环境隔离。

本机装过很多工具以后，系统环境容易变复杂。Node.js 版本、依赖包、系统路径、权限设置，都可能互相影响。Docker 可以把 OpenClaw 放到容器环境里，减少对系统环境的影响。

Docker 也方便迁移。

同一套 Compose 配置，在本地电脑、测试机、云服务器上都可以复用。镜像、环境变量、端口和数据卷配置清楚，换机器也能较快复现。

Docker 适合测试多版本。

开发者可能要同时测试多个 OpenClaw 版本，或者验证某个 Skill、插件、模型配置在不同环境下的表现。容器可以快速创建、停止、删除和重建，适合实验。

Docker 需要一点基础。

你要知道端口映射是什么意思，数据卷挂载在哪里，容器日志怎么查看，环境变量怎么传进去，镜像怎么拉取和更新。遇到问题时，也要能分清 OpenClaw、Docker 网络、权限、镜像、挂载各自出了什么问题。

课堂主线采用本地脚本安装。Docker 放在进阶路线里讲。

对大部分初学者来说，先把 OpenClaw 的基本运行链路跑通更重要。先知道 Gateway、模型配置、入口、Web UI、TUI 怎么工作，再看 Docker 会更稳。

Docker 部署适合几类人：

- 开发者
- 需要隔离环境的服务器用户
- 需要测试多版本的人
- 熟悉容器、端口、日志和环境变量的人

Docker 排查常看四处。

第一，看容器是否启动。用 `docker ps`，或者看 Docker Desktop 里的运行状态。

第二，看端口是否映射。宿主机访问的端口，要正确映射到容器里的服务端口。

第三，看数据目录是否挂载。配置、日志、生成文件和持久化数据，需要落在明确的位置。

第四，看环境变量是否正确。模型 API Key、服务地址、运行模式、时区和其他参数，都要传到容器里。

Docker 用得好，部署会很干净。Docker 基础薄弱时，排查成本会明显增加。

### 六、四种方案快速对比

四种方案放在一起看：

| 部署方案 | 上手门槛 | 控制权 | 长期在线 | 场景 |
| --- | --- | --- | --- | --- |
| 本地部署 | 中 | 强 | 依赖本机 | 课堂跟做、个人学习、本地文件处理 |
| 云服务器部署 | 中 | 强 | 强 | 长期助手、消息机器人、自动推送 |
| 第三方平台托管 | 低 | 中 | 强 | 快速体验、平台用户、课堂展示 |
| Docker 部署 | 中到高 | 强 | 强 | 开发测试、隔离环境、多版本验证 |

上手门槛看第一次跑起来的成本。第三方平台通常最快。本地部署和云服务器要按步骤安装、配置和验证。Docker 还要理解容器概念。

控制权看文件、配置、日志、版本和权限能掌握到什么程度。本地部署、云服务器、Docker 控制权更强。第三方平台把很多底层细节交给平台处理，体验更轻，边界也由平台决定。

长期在线看服务能否持续运行。本地部署依赖个人电脑。云服务器、第三方平台，以及跑在服务器上的 Docker，更适合长期任务。

场景看你真正要完成的任务。学习 OpenClaw，用本地部署。做长期机器人，用云服务器。快速体验和演示，用第三方平台。开发测试和环境隔离，用 Docker。

课堂里先记四句话：

- 本地部署：学习、跟做、本地资料处理
- 云服务器：长期在线、消息机器人
- 第三方托管：快速体验、平台用户
- Docker：隔离环境、开发测试

后面真正选方案时，回到自己的任务：处理什么资料，在哪个入口使用，要在线多久，能接受多少维护成本，能承担哪些权限风险。

### 七、安全与权限边界

部署方案选完，接着看权限边界。

OpenClaw 跑起来以后，会从聊天窗口进入真实工作环境。它可能读取文件、发送消息、调用 API、执行脚本、访问网页、更新文档、触发自动化任务。能力越强，权限越要细。

第一层是消息入口。

飞书、企业微信、钉钉、QQ、Telegram 这些入口，都有私聊、群聊、机器人权限和事件配置。刚开始使用时，先在私聊或测试群里验证。确认消息能接收、能回复、上下文清楚，再进入正式群和团队入口。

群聊要格外谨慎。群里有多人发言，机器人要知道什么时候响应、响应谁、使用哪个上下文。飞书机器人还涉及应用权限、事件订阅、发布版本等配置。权限缺失时，可能出现事件已收到、回复环节卡住，或者入口已配置、用户信息读取失败。

第二层是文件访问。

本地部署时，OpenClaw 很容易接触本机文件。初期只给课程目录、测试目录、项目样例目录这类小范围资料。等大家理解它的行为，再逐步扩大范围。

文件访问看两件事：能读什么，能写什么。

能读，意味着它可能把资料放进任务上下文。能写，意味着它可能修改文件、生成结果、覆盖内容。课程练习会尽量使用专门目录，让所有变化都看得见、能恢复、能复盘。

第三层是工具调用。

工具包括脚本、命令、API、浏览器操作、文档写入、消息发送。这里的风险比普通对话高，因为它会改变外部状态。

读取文件只是获取信息。写文件会改变本地内容。发送飞书消息会影响他人。调用业务 API 可能创建订单、修改数据、触发流程。执行脚本可能批量改文件，也可能访问系统资源。

工具权限从小范围开始。课程里先做可观察、可撤回、影响小的任务。熟悉 OpenClaw 的行为以后，再考虑更复杂的自动化。

第四层是外部系统。

生产系统、客户数据、财务数据、业务后台、公司内部知识库，都属于高敏感区域。接入这类系统时，要单独评估账号权限、日志记录、审批流程、人工确认和回滚方案。

长期助手还要关注 API Key。

模型 API Key、飞书应用密钥、云服务器密码、平台 Token 都要妥善保存。配置文件、环境变量、日志输出、截图和录屏里都可能暴露密钥。上课演示时，密钥和 Token 要打码，个人账号也要隔离。

这部分我建议大家养成几个操作习惯：

- 先试用，再授权
- 先小范围，再扩大
- 先观察，再自动化
- 先测试入口，再进入正式入口
- 先处理样例数据，再处理真实资料

OpenClaw 的价值来自真实工作流。真实工作流里有文件、消息、账号、权限和业务系统。部署跑通只是第一步，权限边界决定它能用多久、用到多深。

下一节进入本地部署实操。大家按课堂主线在自己的电脑上跑起来，完成安装、onboard、模型配置、Web UI 或 TUI 验证。跑通以后，再根据自己的使用目标选择云服务器、第三方托管或 Docker。
