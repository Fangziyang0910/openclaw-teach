# 项目协作说明：OpenClaw 从入门到案例实践

## 本项目核心任务

本项目用于准备一门线下/现场教学课程，课程名称为：**OpenClaw 从入门到案例实践**。

课程时长为一个上午，目标是让学员在有限时间内完成从认识 OpenClaw 到真实上手部署、接入飞书、配置关键能力并完成案例实战的完整体验。

本项目不是普通代码项目，而是一个课程资料与教学交付项目。后续所有修改都应优先服务于课程交付质量，包括讲解逻辑、实操顺序、学员跟做成功率、现场故障兜底方案和案例可演示性。

## 课程目标

课程需要帮助学员完成以下学习目标：

1. 认识 OpenClaw 是什么，理解它能解决哪些真实问题。
2. 理解 OpenClaw 的基本原理、核心架构和运行逻辑。
3. 了解 OpenClaw 的多种部署思路，包括本地部署、云端部署、一键部署、面板化部署等路线。
4. 在课堂现场手把手带学员在自己的电脑上部署 OpenClaw。
5. 带学员将 OpenClaw 接入飞书，让 OpenClaw 成为可日常使用的助手入口。
6. 带学员完成 Skills 配置，让 OpenClaw 具备可扩展能力。
7. 带学员理解并配置定时任务、Soul、Memory 等关键能力。
8. 结合三个案例完成实战演练，让学员理解 OpenClaw 不是玩具，而是可以落地到工作流中的 Agent 工具。

## 课程资料分工

本项目需要同时维护三类课程资料，它们面向的读者和写法不同，后续不要混淆：

1. **讲师讲稿**：给讲师本人备课和授课时参考使用，配合 Slide 逐页讲解。讲稿可以包含口播词、讲解重点、转场话术、提醒事项、现场演示提示、可能的学员问题和故障兜底方案。讲稿不是发给学员的资料，可以更详细、更口语化、更偏授课节奏。
2. **实战手册**：给学员人手一份，用于课堂跟做和课后复习。手册应强调步骤清晰、命令可复制、截图/配置准确、验证方式明确、常见问题可自查。
3. **Slide**：给学员现场观看，用于建立认知、控制课堂节奏和呈现关键图示。Slide 不应堆满讲稿内容，应保持重点突出、视觉清晰、适合投影展示。

后续如果创建讲师讲稿，建议文件命名为 `OpenClaw从入门到案例实践讲师讲稿.md`。讲稿应尽量按 Slide 页码或章节组织，方便讲师授课时对照使用。

## 参考教程库说明

`awesome-openclaw-tutorial/` 是本项目的重要参考资料目录。它包含 OpenClaw 从入门、部署、配置、扩展到实战应用的完整教学资料。

该目录当前主要包含：

1. `README.md`：教程总入口，包含教程定位、目录、版本说明和资源导航。
2. `LEARNING-PATH.md`：7 天学习路径，可用于辅助设计课程节奏。
3. `Openclaw史上最简单教程，小白一键部署.md`：多平台部署路线合集，可用于部署思路讲解。
4. `docs/`：主线教程，共覆盖入门、安装、快速上手、文件管理、知识库、日程、自动化、Skills、多平台、API、高级配置和实战案例。
5. `appendix/`：附录工具资料，包括命令速查、Skills 清单、API 服务商对比、常见问题、配置模板、最佳实践、视频教程、飞书检查清单等。
6. `examples/README.md`：示例说明文档，可用于案例设计参考。
7. `jvsclaw部署.md`、`宝塔面版claw.md` 等：补充部署方案文章，可用于讲解不同部署路线的取舍。

该目录中的 `.git`、`.vscode`、脚本、站点配置、搜索索引、发布工具和示例代码等无关内容已经删除。后续不要默认恢复这些无关文件，除非用户明确要求。

## 课程内容重点

后续围绕本项目工作时，应重点关注以下内容模块：

1. **课程开场**：说明 AI Agent 与 OpenClaw 的价值，降低学员认知门槛。
2. **OpenClaw 原理**：讲清 Agent、模型、工具、Skills、Memory、Soul、定时任务、消息入口之间的关系。
3. **部署路线**：对比本地部署、云端部署、飞书妙搭、JVSClaw、宝塔面板等路线，帮助学员理解为什么课堂选择当前方案。
4. **本地部署实操**：保证步骤清晰、命令准确、错误可排查，适合现场跟做。
5. **飞书接入**：重点关注飞书 Bot 创建、权限配置、回调/事件配置、消息交互验证和常见问题。
6. **Skills 配置**：讲清 Skills 的作用、安装方式、安全注意事项和课堂推荐技能。
7. **定时任务配置**：让学员能理解并完成可运行的自动化任务。
8. **Soul 配置**：帮助学员建立个性化助手设定，使 OpenClaw 更像长期助手。
9. **Memory 配置**：讲清短期记忆、长期记忆、知识沉淀和 Memory Wiki / Active Memory 的关系。
10. **三个案例实战**：案例必须能现场演示、能体现价值、步骤不要过长，并且要有失败兜底方案。

## Skills 使用要求

本项目在制作课程资料时应充分利用可用 Skills，而不是完全凭空编写。

1. **Slide 设计与制作**：涉及 Slide 结构、视觉设计、动画、页面布局、HTML 演示页或最终课件效果时，应使用项目中的 `frontend-slides` Skill，并遵循其工作流。
2. **讲师讲稿编写**：涉及讲师口播词、转场话术、课堂讲解、案例引导、FAQ 回答时，应使用 `humanizer` 类写作润色 Skill，让表达更自然、更像真实讲师授课，而不是 AI 生成腔。
3. **实战手册编写**：涉及学员手册文字、步骤说明、提示语、常见问题解释时，也应使用 `humanizer` 类写作润色 Skill，在保持准确性的基础上降低 AI 味道。
4. **Slide 文案优化**：涉及标题、短句、页面文案、模块导语和总结语时，应使用 `humanizer` 类写作润色 Skill，使文案更口语、更清楚、更适合现场教学。
5. **内容优先级**：先保证技术准确和课堂可执行，再进行 humanizer 润色；不要为了口语化牺牲命令、配置、路径、版本和操作步骤的准确性。
6. **Skill 缺失处理**：如果当前环境暂时没有对应 Skill 文件，也要按该 Skill 的目标执行：Slide 追求高质量视觉与节奏，讲稿/手册/Slide 文案追求自然、有人味、少 AI 腔。

## 教学设计原则

1. 一个上午课程时间有限，内容应以“能讲清、能跑通、能让学员带走”为优先。
2. Slide 负责建立认知和节奏，实战手册负责让学员照着完成操作。
3. 每个实操环节都应包含：目标、前置条件、操作步骤、验证方式、常见问题。
4. 命令、配置、截图和案例说明要尽量一致，避免 Slide 与手册互相矛盾。
5. 复杂概念要用图示、类比和流程拆解，不要只堆术语。
6. 课堂案例应围绕真实工作流设计，例如个人助理、知识管理、自动日报、内容生产、任务提醒等。
7. 对可能出现网络、权限、依赖、平台配置失败的步骤，要准备替代路径或演示兜底。
8. 优先让学员获得“我真的部署成功并能使用”的体验，再拓展高级概念。

## 后续协作建议

当用户要求优化课程资料时，优先按以下顺序处理：

1. 先理解用户当前要改的是 Slide、实战手册、参考资料整理，还是完整课程大纲。
2. 如果修改 Slide，必须优先修改 `src/` 中的源码，然后使用项目已有命令脚本合成最终的 `openclaw-training-ppt.html`。
3. 不要直接把 `openclaw-training-ppt.html` 当作主要编辑入口；它应被视为构建产物，除非用户明确要求直接修补最终 HTML。
4. 如果修改实战手册，优先检查步骤准确性、命令可复制性、环境兼容性和故障排查说明。
5. 如果补充课程内容，优先从 `awesome-openclaw-tutorial/` 中提炼内容，而不是凭空生成。
6. 如果涉及前端 Slide 展示效果，应使用 Playwright MCP 检查页面渲染和交互效果。
7. 如果涉及最新库/API/平台配置，应按项目规则调用相应 MCP 获取官方文档。
8. 如果涉及 Git 操作，不要自动提交，除非用户明确要求。
