# 第一部分：认识 OpenClaw

## 一、从聊天窗口到真实工作流

各位过去接触 AI，大概率都是从一个聊天窗口开始的。

ChatGPT、Claude、DeepSeek、Kimi、通义千问这类工具，大家都不陌生。打开网页或 App，输入一句话，AI 就能回答问题、写文案、总结文章、翻译内容、修改代码、生成方案。对普通用户来说，这已经带来了很明显的效率提升。

比如，写通知，可以让 AI 帮忙润色；写日报周报，可以让 AI 帮忙整理结构；看一篇长文章，可以让 AI 提炼重点；遇到一个陌生概念，可以让 AI 用更简单的方式解释；写代码遇到报错，也可以把错误信息贴进去，让 AI 帮忙分析。

这就是我们最熟悉的 <u>对话型 AI</u>：用户把问题交给 AI，AI 在聊天窗口里生成回答。它很适合问答、写作、总结和辅助思考。

但一放到真实工作里，问题就出来了。

**第一个问题：上下文搬运。**  
真实任务往往带着大量背景信息。比如一份会议纪要，可能要结合会议聊天记录、参会人员、项目背景、历史决策和后续安排。用户要让 AI 完成这个任务，通常得先把这些材料整理好，再复制到聊天窗口里。材料越多，任务越复杂，前面的准备成本就越高。

**第二个问题：结果转移。**  
AI 生成内容以后，用户还要把结果复制回真实工作环境。会议纪要要放回飞书文档，待办要放进任务系统，日报要发到群里，代码要回到项目目录。AI 生成了内容，但后面的落地动作还得人来做。

**第三个问题：任务持续性弱。**  
很多 AI 对话都是一次性的。今天让它总结文章，明天让它写周报，后天让它整理会议记录，每一次都像重新开始。它很难持续关注一个入口里的新消息，也很难按时间自动处理任务，更难围绕一个长期目标持续跟进。

**第四个问题：工作现场分散。**  
真实工作很少只发生在一个聊天窗口里。它可能同时发生在飞书消息、文档、表格、浏览器、本地文件、内部系统、代码仓库、命令行脚本和自动化平台中。AI 如果只待在聊天窗口里，就很难碰到这些现场。

**第五个问题：工具使用受限。**  
很多时候，我们希望 AI 能读取文件、连接外部系统或服务、运行预先写好的自动化操作、发送消息、更新文档、整理表格、触发固定流程。对话型 AI 可以告诉我们应该怎么做，但许多实际动作还要用户自己完成。

所以，真实工作流对 AI 提出了新的要求：

- AI 要出现在用户原本使用的入口里，比如飞书、Telegram、Slack、微信、网页控制台或者团队协作工具。
- AI 要理解文件、消息、历史资料、项目背景和用户偏好。
- AI 要能连接 API、脚本、数据库、网页服务和业务系统。
- AI 还要把结果交付到用户真正使用的位置。

**我们今天讲 OpenClaw，就是从这个变化开始。**

## 二、Agent：从回答问题到执行任务

要理解 OpenClaw，先看一个词：<u>Agent</u>。

Agent 是围绕目标推进任务的 AI 系统。它和普通聊天助手最大的区别，是会使用工具。

普通对话型 AI 更偏向生成回答。用户问一个问题，它回答一个问题；用户给一段材料，它生成一段内容。Agent 会往前多走一步：根据用户目标组织步骤、选择工具、执行动作，再把结果交付到指定位置。

> Agent 是会使用工具、能推进任务的 AI 助手。

它通常有几类能力：

- **目标理解**：用户说“帮我整理一下今天会议的重点”，Agent 要判断用户真正想要的是会议结论、行动项、负责人、截止时间，还是一份可以直接发给团队的会议纪要。
- **任务拆解**：复杂任务通常不能一步完成。比如生成会议纪要，可能要先获取会议材料，再识别主要讨论，再整理决策点，再提取待办事项，最后输出成文档格式。
- **工具选择**：Agent 要判断当前任务要用什么工具。可能是读文件、查网页、调用飞书接口、操作表格、运行脚本，也可能是读取数据库。
- **过程执行**：Agent 可以根据中间结果继续行动。比如它读取了一份会议记录，发现里面涉及多个项目，就继续按项目拆分待办；发现某个负责人缺失，就把它标记为待确认。
- **上下文利用**：Agent 可以结合历史信息、用户偏好、项目背景和当前任务进行判断。比如用户长期希望输出“结论在前、行动项在后”的格式，Agent 就可以按照这个偏好组织结果。
- **结果交付**：结果最终要回到用户真正使用的位置。可能是飞书消息、飞书文档、表格、邮件、项目系统，也可能是本地文件。

我们用会议纪要举例。

在传统对话型 AI 中，用户通常先复制会议内容，让 AI 生成纪要，再把结果复制回飞书或文档。这个流程里，AI 主要负责生成文本。

在 Agent 模式中，流程更接近真实工作：Agent 从飞书或指定入口接收任务，获取会议内容，整理会议结论、待办事项、负责人和截止时间，再将结果返回飞书或写入文档。任务从一次文本生成，变成了**获取材料、整理结构、提炼待办、交付结果**的一组动作。

再看出差计划。

对话型 AI 可以根据用户输入，生成一份出差计划文本。Agent 可以进一步读取日程、查询交通和酒店信息、整理出差待办、生成计划，并发送到指定入口。

这类任务都有几个共同点：**步骤多，信息来源多，要用工具，而且有明确的交付目标。**

真实工作流里的任务，经常就是这样一串动作：找资料，判断资料价值，整理结构，生成内容，调整格式，发送给指定对象，归档结果，后续继续跟进。

Agent 的价值，就在于把这些动作组织起来，让 AI 从回答问题，逐步走向执行任务。

## 三、OpenClaw 是什么

理解了 Agent，再来看 OpenClaw。

OpenClaw 的出现，可以从一个很朴素的需求说起：**AI 要进入真实工作现场。**

真实工作分散在很多地方。我们在飞书里沟通，在文档里写材料，在表格里整理数据，在浏览器里查资料，在本地目录里保存文件，在脚本和自动化工具里处理重复任务。一个真正有用的 AI 助手，最好能靠近这些地方。

OpenClaw 的创始人 Peter Steinberger 介绍过，这个项目最早来自一个周末项目，后来经历了命名变化，并在社区中快速扩散。官方博客里也记录了从 Clawd、Moltbot 到 OpenClaw 的命名过程。

从定位上看，OpenClaw 是一个 <u>自托管的个人 AI 助手系统</u>。简单说，就是它可以运行在用户自己的电脑或服务器上。它通过 Gateway 把聊天入口、模型能力、Agent、Skills 和工具连接起来。这里的 Gateway，可以先理解成 OpenClaw 的**中转和调度服务**：消息从飞书、Telegram 这类入口进来，Gateway 负责接住，再转给后面的 AI 助手处理。

这句话听起来有点技术化，我们换一种说法：

- **大模型是大脑**，负责理解、推理和生成。
- **OpenClaw 是调度系统**，负责接收消息、安排任务、连接各种能力。
- **消息入口是使用界面**，用户可以通过飞书、Telegram、Slack、Web UI 等入口和助手对话。
- **Skills 是工具箱**，也可以理解为教 Agent 使用某种能力的说明书和配套资源。
- **本地文件、脚本、API 和自动化任务，是 AI 实际工作的环境。**

OpenClaw 做的事情，就是把这些部分组织成一个个人 AI 助手系统。

所以，我们后面的课程会围绕四件事展开：**部署、飞书接入、Skills 配置、案例实践。**

部署解决系统怎么跑起来。飞书接入解决 AI 怎么来到工作入口。Skills 配置解决能力怎么扩展。案例实践则用真实任务看看它能帮我们做到什么程度。

## 四、OpenClaw 为什么适合真实工作流

OpenClaw 适合真实工作流，可以从三个角度理解：**入口价值、能力价值、持续价值。**

### 1. 入口价值：在用户原本的地方使用 AI

很多工作任务本来就从消息入口产生。

团队在飞书群里讨论项目进度，老师在群里通知课程安排，运营人员在群里同步活动方案，项目负责人在群里安排任务，学生在群里提问。消息入口本身就是一个高频工作入口。

官方文档中提到，OpenClaw 的 Gateway 可以接入多种消息入口。除了 WebChat 和移动端入口，也支持 Discord、Google Chat、iMessage、Matrix、Microsoft Teams、Signal、Slack、Telegram、WhatsApp、Zalo 等工具。

这门课选择飞书作为主要实操入口，因为飞书在国内团队协作场景中很常见，而且同时具备消息、文档、表格、日历、任务等能力。OpenClaw 的 Feishu 文档中也说明，飞书入口通过机器人接收消息变化，当前版本已经内置 Feishu 插件，无需单独安装。

对用户来说，这意味着不用每次都打开一个单独的 AI 网站。比如在飞书里说一句：“帮我把这段讨论整理成会议纪要。”系统就可以围绕这个入口继续处理。

### 2. 能力价值：把模型能力连接到文件、工具和系统

只接入消息入口还不够。AI 要完成任务，还需要能使用资源。

这些资源包括本地文件、本地目录、脚本、API、数据库、网页服务、自动化平台和业务系统。OpenClaw 的 Gateway 负责把入口、Agent、模型和工具组织起来。官方架构文档中提到，一个长期运行的 Gateway 会管理不同的消息入口，命令行、Web UI、自动化任务和其他节点都可以连接到它。

在这里，OpenClaw 更像一个<u>中间调度层</u>。用户从飞书发出任务，OpenClaw 接收任务，把任务交给 Agent。Agent 结合模型判断要用什么能力，再调用文件、工具、Skills 或外部服务。

Skills 是这里很重要的扩展机制。官方 Skills 文档说明，OpenClaw 通过一个个 Skill 文件夹教会 Agent 如何使用工具。每个 Skill 都是一个目录，里面最重要的是 `SKILL.md` 文件，写着这个能力什么时候用、怎么用。

换成更口语的说法，**Skill 像一份能力说明书。** 它告诉 Agent：什么时候用这个能力，怎么用，需要什么输入，能输出什么结果。

有了 Skills，OpenClaw 就能从通用助手扩展到具体工作场景。比如文件整理助手、飞书通知助手、网页监控助手、课程资料助手、日报周报助手、信息检索助手。

### 3. 持续价值：把一次性问答沉淀为长期助手

真实工作经常具有持续性。

比如每天早上收集行业新闻，下午生成课程资料，晚上总结当天工作；每周生成项目周报；每次群里开完会后自动整理会议纪要；每次某个信息源更新后自动推送提醒。

OpenClaw 可以长期运行在个人电脑或服务器上，通过 Gateway 管理会话、入口和工具。官方文档中也提到，Gateway 负责统一管理会话、消息分发和入口连接，控制界面、命令行和 Web UI 都可以连接到它。

长期助手还涉及记忆和上下文。OpenClaw 的 memory 文档中提到，Agent 有 `MEMORY.md`、每日 notes 和可选的 `DREAMS.md` 等记忆相关文件。其中 `MEMORY.md` 可以理解为记忆文件，用来记录比较稳定的偏好、事实和决策。

所以，OpenClaw 这类系统更适合做长期助手。它不只回答今天这一个问题，还可以慢慢熟悉用户的工作方式，帮用户处理反复出现的信息任务。

## 五、OpenClaw 能用来干什么

接下来，我们用几个具体场景来理解 OpenClaw 的用途。

### 1. 消息与协作助手

第一类场景，是消息与协作助手。也就是把 OpenClaw 放进飞书这样的消息入口，让它围绕群聊、会议和项目同步来帮忙。

用户可以直接在飞书里提问：“帮我把这段讨论整理成会议纪要。”也可以说：“根据刚才的聊天内容，整理出三个行动项。”还可以说：“帮我写一段项目进展同步，语气正式一点，适合发到团队群。”

这里的价值很直观：**材料在消息入口里，AI 也在消息入口里，结果仍然回到消息入口里。**

飞书插件文档里列出了很多能力，包括消息读取与发送、文档创建与更新、表格编辑、日历事件管理和任务管理等。也就是说，飞书接入不是简单多了一个聊天框，它让 AI 有机会接触团队协作里的消息、文档、表格、日历和任务。

所以这一类场景可以覆盖即时问答、群聊总结、会议纪要、项目同步、通知生成和行动项整理。需要注意的是，这些能力通常依赖飞书接入、权限配置和相应工具，不是装完 OpenClaw 就自动拥有全部能力。

### 2. 资料整理与内容生成

第二类场景是资料整理。

比如我们准备一门课，手头可能有课程大纲、教学案例、产品文档、参考文章、实操流程、常见问题。在提供资料或配置资料访问权限后，OpenClaw 可以帮助把这些资料整理成课程结构、讲稿、讲义、案例脚本和课堂练习。

再比如团队准备一次项目汇报，可能要汇总群聊讨论、历史文档、项目里程碑、数据表格和负责人反馈。在资料准备充分的情况下，OpenClaw 可以辅助整理出汇报大纲、项目进展、风险点和下一步计划。

这类场景的重点，是把零散材料整理成结构化内容。输出可以是大纲、表格、清单、报告、会议纪要、课程笔记，也可以是行动项。

对普通学员来说，这类场景最容易上手，因为它和大家已经熟悉的 AI 使用方式接近。不同之处在于，OpenClaw 更适合把资料来源、输出位置和重复流程串起来。

### 3. 本地文件与个人知识助手

第三类场景，是本地文件与个人知识助手。

很多人的资料都放在本地目录里：项目文件、课程资料、学习笔记、论文材料、合同模板、脚本文件、图片素材、历史报告。普通 AI 工具通常需要用户主动上传或复制内容。OpenClaw 这类自托管助手系统，更适合围绕本地环境组织工作。

在配置本地权限和相应工具后，用户可以让它帮忙定位某个项目文件，总结某个目录下的文档，整理一批材料的主题，或者根据项目资料生成说明文档。

OpenClaw 的 Agent 运行文档中提到，它会使用一个固定的工作目录来放工具和上下文；新会话启动时，还会加载一组用户可以编辑的启动配置文件，例如 `AGENTS.md`、`SOUL.md`、`TOOLS.md`、`IDENTITY.md`、`USER.md` 等。这些文件可以告诉助手如何工作、用户偏好是什么、能用哪些工具。

这意味着，OpenClaw 有自己的工作空间，可以围绕用户配置、工具说明、角色设定和本地上下文运行。它会在一个配置好的工作空间里处理任务，而不是只等用户发一句话。

本地文件能力再往前走一步，就会变成个人知识助手。比如一名教师可以用它积累课程材料，一名研究人员可以用它整理论文笔记，一名运营人员可以用它维护活动资料，一个小团队可以用它沉淀项目经验。这里同样要依赖资料、权限、记忆和规则配置。

### 4. 自动化与长期助手

第四类场景，是自动化与长期助手。

比如每天早上自动收集行业新闻，整理成摘要后发送到飞书；每周五生成项目周报；定时检查某个网页是否更新；每天把学习记录整理成清单；把一批材料定期归档到指定目录。

这类任务的特点是重复、固定、有明确输出位置。它们很适合沉淀成自动化流程。

OpenClaw 的配置文档中提到，常见配置包括连接消息入口、控制谁可以给机器人发消息、设置模型和工具、配置权限隔离和自动化任务，以及调整会话、媒体、网络或界面相关选项。

换句话说，OpenClaw 可以围绕入口、模型、工具和自动化配置，长期运行起来。普通用户可以先从简单任务开始，比如每天生成资讯摘要，或者每次群聊讨论后生成待办清单。

长期助手的价值，在于积累用户的偏好、背景和历史资料。OpenClaw 的 memory 文档中，`MEMORY.md` 用于保存长期记忆，日记文件用于记录当天和前一天的上下文。

可以简单理解：**一次性 AI 工具像临时帮手，长期助手像熟悉你工作方式的助理。**

如果把它放到团队协作里，还要特别注意权限。飞书插件文档明确提醒，授权后 OpenClaw 会在授权范围内以用户身份行动，可能带来敏感数据泄露或未授权操作等风险；文档也建议把连接 OpenClaw 的飞书机器人作为私人会话助手使用，谨慎加入群聊。

所以后面实操时，我们会先用私聊场景跑通，再扩展到群聊；先开必要权限，再增加高级能力。

## 六、为什么社区关注 OpenClaw

OpenClaw 受到社区关注，可以从项目发展、用户需求、开源生态和传播效应几个角度看。

### 1. 项目发展时间线

OpenClaw 的故事很有传播性。

官方博客中，Peter Steinberger 写到，这个项目最早来自一个周末项目，最初曾叫 WhatsApp Relay，后来在两个月内获得超过 100,000 个 GitHub stars，并在一周内吸引了 200 万访问者。官方博客也记录了 Clawd、Moltbot、OpenClaw 之间的命名变化。

它的传播速度，和一个非常现实的需求有关：用户希望 AI 能出现在每天使用的消息入口和工作环境里。

官方 README 也写到，OpenClaw was built for Molty，由 Peter Steinberger 和社区共同推动。

这个发展过程可以按几个阶段看：

- **个人原型阶段**：项目从“让 AI 进入消息入口”的想法开始。
- **Clawd 阶段**：项目名称带有 Claude 和 claw 的双关，开始获得早期关注。
- **Moltbot 阶段**：社区参与命名和传播，项目形态继续扩大。
- **OpenClaw 阶段**：项目以更开放和稳定的名称传播，围绕部署、入口接入、Skills 和案例实践出现了大量讨论。
- **相关实验阶段**：Moltbook 等 Agent 社区实验继续带动个人 AI 助手、Agent Gateway 和 AI 社区形态的讨论。

这里要分清楚：**OpenClaw 是个人 AI 助手和 Gateway 系统，Moltbook 是围绕 Agent 玩法产生的外部实验和传播现象。**

### 2. 社区关注的原因

OpenClaw 的社区关注，来自几个具体需求：

- **个人 AI Agent 需求**：用户希望 AI 能进入飞书、Telegram、Slack、WhatsApp 等常用入口，长期可用，处理日常任务。
- **自托管与可控性**：OpenClaw 官方文档把 self-hosted 作为特点之一。也就是说，用户可以把它运行在自己的电脑或服务器上，按自己的规则管理环境。
- **多入口接入**：同一个 Gateway 可以连接多个消息入口，用户有机会在不同聊天工具中访问同一个助手。官方架构文档也提到，一个长期运行的 Gateway 可以管理 WhatsApp、Telegram、Slack、Discord、Signal、iMessage、WebChat 等入口。
- **Skills 扩展**：ClawHub 官方文档把 ClawHub 定义为 OpenClaw skills 和 plugins 的公共技能仓库。用户可以用原生 `openclaw` 命令搜索、安装和更新 skills。
- **工作流落地**：OpenClaw 更适合围绕资料整理、自动推送、群聊总结、知识库和团队协作组织完整流程。

### 3. 中文社区关注方向

中文社区关注 OpenClaw，主要集中在几个方向：

- **部署实践**：本地部署、服务器部署、Docker 部署、环境配置、Node 版本、Gateway 状态检查、常见报错处理。
- **入口接入**：飞书是国内很典型的团队协作入口，因此飞书接入、机器人配置、权限设置、消息收发、群聊使用，成为中文教程的重点。
- **Skills 分享**：用户会关注哪些 Skills 能处理文件、检索信息、搭建知识库、生成媒体内容、推送资讯、处理飞书任务。
- **“养虾”文化**：OpenClaw 的吉祥物和社区表达带有 lobster 文化，很多用户把配置、调试、训练和长期使用个人 AI 助手称为“养虾”。这个说法很有社区感，也让一个技术项目更容易传播。
- **案例实践**：比如个人知识库、自动资讯推送、群聊总结、日报周报、课程助手、团队协作助手等。相比单纯部署教程，这些案例更能说明 OpenClaw 在真实任务里能怎么用。

### 5. 外部传播与生态实验

OpenClaw 的外部传播，和个人 AI 助手的发展方向有关。

过去很多 AI 产品以单一聊天界面为主。OpenClaw 展示的是另一条路线：用户把助手部署在自己的机器上，连接常用入口，配置自己的工具和 Skills，让 AI 更贴近个人工作习惯。

Moltbook 等 Agent 社区实验，也推动了关于 AI 社区、AI 助手人格化、Agent 组织方式的讨论。部署教程、飞书接入教程、Skills 分享和案例视频，也让更多普通用户开始上手尝试。

所以，认识 OpenClaw 时，不要只看它的开源热度。更重要的是看到那些具体案例：群聊总结、资讯推送、知识库、日报周报、课程助手。正是这些场景，让 OpenClaw 更容易被普通用户理解。
