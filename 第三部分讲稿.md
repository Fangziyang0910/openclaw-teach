# 第三部分：OpenClaw 的部署思路

## 一、部署方案先看使用目标

OpenClaw 可以跑在本地电脑上，也可以跑在云服务器上，还可以通过第三方平台直接托管。Docker 也是一种常见部署方式，适合想把环境隔离起来的人。

讲部署方案，先看三个问题。

第一个问题：**OpenClaw 要在哪里干活。**

如果主要处理本机文件、课程资料、个人项目目录，本地部署最直接。文件就在电脑里，OpenClaw 也在电脑里，读取、生成、验证都比较顺手。

如果主要接飞书、企业微信、钉钉、QQ 这类消息入口，而且希望机器人长期在线，云服务器会更合适。服务器可以一直开着，公网访问也更稳定。

如果只是想快速体验，先看 OpenClaw 长什么样、能做什么，第三方平台托管最快。很多平台已经准备好了运行环境，用户只要按页面流程创建实例、配置账号或额度，就可以开始用。

如果有开发测试、版本隔离、迁移复现的需求，Docker 更方便。它把 OpenClaw 和系统环境隔开，部署、升级、清理都更可控。

第二个问题：**OpenClaw 要在线多久。**

课堂跟做、个人学习、临时体验，本地电脑开着就够了。学员只要完成安装、onboard、模型配置和基础验证，就能理解 OpenClaw 的主流程。

长期机器人、定时任务、自动推送、消息监听，需要稳定在线。这个时候云服务器或托管平台的优势会更明显。因为它们可以脱离个人电脑的开关机状态，持续接收消息和执行任务。

第三个问题：**OpenClaw 能碰到哪些资料和能力。**

OpenClaw 一旦跑起来，就可能接触文件、消息、API Key、脚本、浏览器、自动化任务和外部系统。部署位置越接近真实业务，权限边界就越要清楚。

所以部署方案的选择，实际是在选择三件事：运行位置、在线方式、权限边界。

这门课的主线采用本地部署。原因很简单：每个学员都可以在自己的电脑上完成安装、配置、验证，马上看到 OpenClaw 怎么运行。云服务器、第三方托管和 Docker 会放在部署思路里一起讲，方便大家课后按自己的场景选路线。

---

## 二、本地部署

本地部署，就是把 OpenClaw 安装在自己的电脑上运行。

macOS、Windows、Linux 都可以部署。常见流程是：先准备系统环境，再运行安装脚本，然后执行 onboard，配置模型，最后打开 Web UI 或 TUI 做验证。

这里的 onboard，可以理解成第一次启动时的引导配置。它会带你完成基础设置，让 OpenClaw 知道接下来用什么模型、怎么启动、入口在哪里、基础配置是否完整。

Web UI 是网页界面，适合大部分学员。它更直观，可以看到配置、状态和交互入口。TUI 是终端界面，适合经常使用命令行的人，尤其适合服务器环境和远程运维场景。

本地部署最大的特点，是控制权强。

OpenClaw 的文件、配置、日志、运行状态都在自己的电脑上。出了问题，排查路径也比较清楚：先看安装是否完成，再看 Gateway 是否启动，再看模型配置是否生效，再看 Web UI 或 TUI 是否能打开。

它也方便处理本地资料。

比如这门课的资料就在一个项目目录里，有大纲、讲稿、实战手册、Slide 源码、图片、参考教程。OpenClaw 如果运行在本机，就更容易围绕这些本地文件干活。让它整理课程目录、生成讲稿、检查文档、修改文件，路径都很直接。

本地部署也很适合课堂。

课堂要让每个学员真正上手，最重要的是让大家经历完整流程：安装、启动、配置模型、打开入口、发出第一条请求、看到系统正常回复。这个过程能让学员理解 OpenClaw 的基本运行方式。

本地部署也有几个现实条件。

电脑要保持在线。电脑休眠、关机、网络断开，OpenClaw 就会停止工作。系统权限也会影响使用，尤其是读写文件、执行命令、访问本地目录时，权限配置要谨慎。

还有一个安全点要讲清楚。

本地部署建议放在专门的工作目录、测试目录或闲置设备上。给 OpenClaw 的文件访问范围要从小开始。课程里我们用课程目录和演示文件做练习，这样更容易观察行为，也方便恢复。

适合本地部署的人群很明确：

- 课堂学员。
- 个人学习用户。
- 想处理本地文件和项目资料的人。
- 想理解 OpenClaw 运行原理的人。

本地部署学会以后，再去看云服务器、托管平台和 Docker，会轻松很多。因为底层逻辑是一样的：OpenClaw 要启动 Gateway，要配置模型，要有入口，要能接收任务，要能返回结果。

## 三、云服务器部署

云服务器部署，就是把 OpenClaw 放到云厂商提供的服务器上运行。

常见选择包括腾讯云 Lighthouse、火山引擎、阿里云轻量服务器等。也可以使用自己已经购买的云主机。部署方式可以是平台镜像、一键脚本，也可以是手动安装。

云服务器的主要价值，是长期在线。

本地电脑适合学习和本地资料处理。云服务器适合让 OpenClaw 变成一个长期运行的助手。比如飞书机器人、企业微信机器人、QQ 机器人、定时收集信息、每天自动整理推送，这类任务都需要持续运行。

消息入口也更稳定。

接飞书时，经常会涉及事件订阅、回调地址、应用权限和消息回复。服务器有公网地址，飞书这类平台更容易把消息送到 OpenClaw。后面做飞书接入时，大家会看到：一个消息机器人要正常工作，需要入口、权限、回调、Gateway 状态都配合起来。

云服务器还有一个好处：多设备访问方便。

OpenClaw 跑在服务器上，你可以从电脑访问，也可以从手机消息入口访问。它跟某一台个人电脑的开关机状态绑定得更少，更像一个长期在线的个人服务。

但是云服务器也要维护。

服务器要看配置、地域、网络、端口、安全组、系统版本、日志、备份和升级。端口开放要谨慎，后台管理入口要有认证，API Key 要保管好，日志里也可能出现敏感信息。

课堂上讲云服务器，重点放在判断框架上。

如果你的目标是课堂学习、本机文件处理、个人体验，本地部署就够用。

如果你的目标是让 OpenClaw 长期接收消息、每天自动运行任务、成为团队入口里的机器人，云服务器更合适。

如果已经有服务器，部署成本主要是时间和维护精力。准备购买服务器时，要把服务器费用、运维成本和安全责任算进去。

适合云服务器部署的人群：

- 想做长期个人助手的人。
- 需要消息机器人长期在线的小团队。
- 有定时收集、整理、推送任务的人。
- 已经熟悉云服务器基础操作的人。

云服务器部署完成以后，排查问题通常按这个顺序看：服务器是否在线，OpenClaw 是否启动，端口是否可访问，模型配置是否正确，消息平台回调是否到达，权限是否完整，日志里是否有错误。

这个顺序很实用。后面做飞书接入时，遇到消息接收失败、机器人回复失败、权限报错，基本都能放回这条链路里定位。

## 四、第三方平台托管

第三方平台托管，就是由平台提供 OpenClaw 的运行环境。

用户通常在网页里创建实例，选择模型或额度，按平台流程接入入口，然后直接开始使用。常见平台包括飞书妙搭、扣子 OpenClaw、Kimi OpenClaw、WorkBuddy、JVSClaw、火山 ArkClaw、智谱 AutoClaw 等。

这类方案最大的特点，是上手快。

很多平台已经准备好了服务器、运行环境、入口集成和基础配置。用户省去 Node.js、脚本安装、端口、进程管理这些前置处理，只要按平台步骤点击创建，就能看到一个能对话、能操作的 OpenClaw 或类 OpenClaw 入口。

它适合快速体验。

比如一个部署经验为零的用户，想先看看 OpenClaw 是否能帮自己做事。第三方平台可以让他绕开环境配置，先体验消息入口、模型回复、技能市场、浏览器操作、文档处理等能力。

它也适合已经使用对应平台的用户。

已经在用飞书的人，可能会优先看飞书妙搭。已经在用 Kimi、扣子、WorkBuddy 或其他平台的人，可能会优先从熟悉的平台入口开始。因为账号、入口、模型、文档、权限体系已经在那个平台里，体验会更顺。

第三方托管也要看边界。

平台会决定可用模型、额度、价格、Skills 市场、工具权限、沙箱能力、数据保存策略、日志可见性和导出能力。对普通用户来说，这些内容一开始感觉很远。真要把它放进工作流里，就要认真看。

比如，你要处理个人学习资料，平台托管很方便。你要处理公司内部资料、客户信息、财务数据、业务后台，就要看平台的数据政策、权限设计和合规要求。

第三方托管的选择标准，可以看五项：

- **入口**：是否已经接好飞书、浏览器、移动端、桌面端或 IM 工具。
- **模型**：是否支持你想用的模型，额度和价格是否能接受。
- **工具**：是否支持文件、网页、文档、表格、浏览器、自动化等能力。
- **数据**：数据保存在哪里，日志怎么看，资料能否导出。
- **边界**：权限怎么控制，沙箱怎么隔离，敏感操作是否需要确认。

适合第三方平台托管的人群：

- 想快速体验 OpenClaw 的人。
- 已经在使用飞书、Kimi、扣子、WorkBuddy 等平台的人。
- 想减少服务器维护投入的人。
- 需要课堂展示或快速演示的人。

课堂里可以把第三方托管理解为“最快看到效果”的路线。本地部署理解为“最适合理解原理和掌握细节”的路线。云服务器理解为“适合长期在线”的路线。

选方案时，别只看创建速度。还要看后续能否接入自己的真实工作，能否控制权限，能否承担费用，能否把资料和结果拿出来。

## 五、Docker 部署

Docker 部署，就是把 OpenClaw 跑在容器里。

容器可以理解成一个相对独立的运行环境。OpenClaw 需要的依赖、服务、端口、环境变量、数据目录，都可以放进一套容器配置里管理。常见方式是使用 Docker Compose，把多个配置写在一起，一条命令启动服务。

Docker 的优势是环境隔离。

本机装过很多工具以后，系统环境可能比较复杂。Node.js 版本、依赖包、系统路径、权限设置，都可能互相影响。Docker 可以把 OpenClaw 放到容器环境里，减少对系统环境的影响。

Docker 也方便迁移。

同一套 Compose 配置，在本地电脑、测试机、云服务器上都可以复用。只要镜像、环境变量、端口和数据卷配置清楚，换机器也能较快复现。

Docker 还适合测试多版本。

开发者可能要同时测试多个版本的 OpenClaw，或者验证某个 Skill、插件、模型配置在不同环境下的表现。容器可以快速创建、停止、删除和重建，适合这种实验性场景。

但是 Docker 需要一些基础。

你要知道端口映射是什么意思，数据卷挂载在哪里，容器日志怎么查看，环境变量怎么传进去，镜像怎么拉取和更新。遇到问题时，也要能分清是 OpenClaw 的问题，还是 Docker 网络、权限、镜像或挂载的问题。

课堂主线采用本地脚本安装。Docker 放在进阶路线里讲。

原因很现实：对大部分初学者来说，先把 OpenClaw 的基本运行链路跑通更重要。先知道 Gateway、模型配置、入口、Web UI、TUI 怎么工作，再去学 Docker 会更稳。

适合 Docker 部署的人群：

- 开发者。
- 需要隔离环境的服务器用户。
- 需要测试多版本的人。
- 熟悉容器、端口、日志和环境变量的人。

Docker 部署时，最常见的排查点有四个。

第一，容器是否启动。看 `docker ps` 或 Docker Desktop 里的运行状态。

第二，端口是否映射。宿主机访问的端口，要正确映射到容器里的服务端口。

第三，数据目录是否挂载。配置、日志、生成文件和持久化数据，需要落在明确的位置。

第四，环境变量是否正确。模型 API Key、服务地址、运行模式、时区和其他参数，都要传到容器里。

Docker 用得好，部署会很干净。Docker 基础薄弱时，排查成本会明显增加。

## 六、四种方案快速对比

这四种方案，可以用一张表快速放在一起看。

| 部署方案 | 上手门槛 | 控制权 | 长期在线 | 适合场景 |
| --- | --- | --- | --- | --- |
| 本地部署 | 中 | 强 | 依赖本机 | 课堂跟做、个人学习、本地文件处理 |
| 云服务器部署 | 中 | 强 | 强 | 长期助手、消息机器人、自动推送 |
| 第三方平台托管 | 低 | 中 | 强 | 快速体验、平台用户、课堂展示 |
| Docker 部署 | 中到高 | 强 | 强 | 开发测试、隔离环境、多版本验证 |

这张表里有几个判断点。

**上手门槛**看的是第一次跑起来的成本。

第三方平台托管通常最快。本地部署和云服务器需要按步骤安装、配置和验证。Docker 要额外理解容器概念。

**控制权**看的是文件、配置、日志、版本和权限能掌握到什么程度。

本地部署、云服务器、Docker 都有较强控制权。第三方平台把很多底层细节交给平台处理，体验更轻，边界也更多取决于平台。

**长期在线**看的是服务是否能持续运行。

本地部署依赖个人电脑。云服务器、第三方平台和 Docker 在服务器上运行时，更适合长期在线任务。

**适合场景**看的是你真正要完成什么。

学习 OpenClaw，本地部署很好。做长期机器人，云服务器很好。快速体验和演示，第三方平台很好。做开发测试和环境隔离，Docker 很好。

课堂里大家可以先记住这四句话：

- 本地部署：适合学习、跟做和本地资料处理。
- 云服务器：适合长期在线和消息机器人。
- 第三方托管：适合快速体验和平台用户。
- Docker：适合隔离环境和开发测试。

后面真正选择方案时，先回到自己的任务：要处理什么资料，要在哪个入口使用，要在线多久，能接受多少维护成本，能承担哪些权限风险。

## 七、安全与权限边界

部署方案选完以后，最重要的是权限边界。

OpenClaw 跑起来以后，它会从聊天窗口进入真实工作环境。它可能读取文件、发送消息、调用 API、执行脚本、访问网页、更新文档、触发自动化任务。能力越强，权限越要清楚。

安全边界可以分成四层看。

第一层是消息入口。

飞书、企业微信、钉钉、QQ、Telegram 这些入口，都有私聊、群聊、机器人权限和事件配置。刚开始使用时，建议先在私聊或测试群里验证。确认消息能接收、能回复、上下文保持清晰，再进入正式群和团队入口。

群聊尤其要谨慎。群里有多人发言，机器人要知道什么时候应该响应、响应谁、使用哪个上下文。飞书机器人还涉及应用权限、事件订阅、发布版本等配置。权限缺失时，可能出现事件已经收到、回复环节卡住，或者入口已经配置、用户信息读取失败的情况。

第二层是文件访问。

本地部署时，OpenClaw 很容易接触本机文件。初期建议只给课程目录、测试目录、项目样例目录这类小范围资料。等大家理解了它的行为，再逐步扩大范围。

文件访问要关注两件事：能读什么，能写什么。

能读，意味着它可能把资料放进任务上下文。能写，意味着它可能修改文件、生成结果、覆盖内容。课程里做练习时，我们会尽量使用专门目录，让所有变化都能看见、能恢复、能复盘。

第三层是工具调用。

工具包括脚本、命令、API、浏览器操作、文档写入、消息发送。这里的风险比普通对话高，因为它会改变外部状态。

比如读取文件只是获取信息。写文件会改变本地内容。发送飞书消息会影响他人。调用业务 API 可能创建订单、修改数据、触发流程。执行脚本可能批量改文件，也可能访问系统资源。

所以工具权限要从小范围开始。课程里先做可观察、可撤回、影响小的任务。等大家熟悉 OpenClaw 的行为，再考虑更复杂的自动化。

第四层是外部系统。

生产系统、客户数据、财务数据、业务后台、公司内部知识库，这些都属于高敏感区域。接入这类系统时，要单独评估账号权限、日志记录、审批流程、人工确认和回滚方案。

长期助手还要关注 API Key。

模型 API Key、飞书应用密钥、云服务器密码、平台 Token 都要妥善保存。配置文件、环境变量、日志输出、截图和录屏里都可能暴露密钥。上课演示时，密钥和 Token 要打码，个人账号也要注意隔离。

这部分要形成一个操作习惯：

先试用，后授权。

先小范围，再扩大。

先观察，再自动化。

先测试入口，再进入正式入口。

先处理样例数据，再处理真实资料。

OpenClaw 的价值来自它能进入真实工作流。真实工作流里有文件、消息、账号、权限和业务系统。部署成功只是第一步，权限边界决定它是否能长期稳定地用下去。

这节课后面进入本地部署实操。大家先按课堂主线在自己的电脑上跑起来，完成安装、onboard、模型配置、Web UI 或 TUI 验证。跑通以后，再根据自己的使用目标选择云服务器、第三方托管或 Docker。
